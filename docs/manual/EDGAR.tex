\begin{abstract}
\noindent
\texttt{EDGAR.jl} pulls company data from the U.S.\ Securities and Exchange
Commission's \href{https://www.sec.gov/search-filings/edgar-application-programming-interfaces}{EDGAR}
system into Julia: a filer's \emph{submission history}, the \emph{XBRL financial facts} extracted from its filings, cross-filer \emph{frames}, with \emph{full-text search} of filing contents. 
\end{abstract}

\section{Introduction}
\label{sec:intro}
EDGAR (\emph{Electronic Data Gathering, Analysis, and Retrieval}) is the U.S. Securities and Exchange Commission's electronic filing system. All publicly listed companies in the United States submit their regulatory disclosures through EDGAR. Since 2009, financial statements within these filings have been tagged using XBRL (eXtensible Business Reporting Language), enabling structured, machine-readable access to financial data.

The SEC provides public access to this information through open JSON REST APIs that do not require registration or API keys. To ensure fair access, the SEC imposes a rate limit of approximately ten requests per second. The caching layer in \texttt{EDGAR.jl} (\S\ref{sec:caching}) helps ensure that usage remains well below this threshold.

\texttt{EDGAR.jl} wraps these endpoints in a lightweight, type-stable API and returns standard Julia objects such as \texttt{NamedTuple}s and \texttt{JSON3} structures. These can be directly integrated with \texttt{Tables.jl}, \texttt{TimeSeries.jl}, or database backends. The package supports three main categories of functionality:

\begin{itemize}[nosep]
  \item \textbf{Filings:} access and process SEC submissions, including listing a filer’s documents, downloading filing packages, converting filings to text, and extracting structured sections such as \emph{Financial Statements} or \emph{Management's Discussion and Analysis}.

  \item \textbf{XBRL Financial Data:} retrieve structured financial facts, including all reported data for a filer (\texttt{company\_facts}), time series for a single concept (\texttt{company\_concept}), or cross-sectional data for a concept across all filers for a given period (\texttt{xbrl\_frames}).

  \item \textbf{Full-Text and CIK Search:} query the contents of filings since 2001 using keyword search via (\texttt{filings\_by\_text}) or CIK codes via (\texttt{filings\_by\_cik}).
\end{itemize}

\medskip
\noindent Beyond these API wrappers, \texttt{EDGAR.jl} adds an interactive
extraction-to-analysis pipeline (Table~\ref{tab:at-a-glance}): \emph{pick} a statement in a
rendered filing --- or bulk-extract a whole filing with no browser --- \emph{export} it four ways,
build a queryable fact \emph{warehouse}, \emph{enrich} it from the filing's own XBRL linkbases,
and \emph{analyse} it as a stitched, cross-period statement. Every fact is traceable back to the
exact region it came from, and the extracted values are validated against the SEC's
\texttt{companyfacts} API.

\section{Installation}
\label{sec:install}

\subsection{Installing from GitHub}
\texttt{EDGAR.jl} requires \textbf{Julia 1.12 or later}. It is not yet registered, so install it straight from GitHub,

\begin{lstlisting}
julia> using Pkg
julia> Pkg.add(url = "https://github.com/Trumpingtons/EDGAR.jl")
\end{lstlisting}

\subsection{Setting an User-Agent}\label{sec:ua}

The SEC's fair-access policy requires each request to include a \texttt{User-Agent} header identifying the requester with a name and a contact email address. Requests that do not include this header are rejected with an HTTP 403 error. There is no API key or registration process, users are simply expected to identify themselves honestly when accessing the data.

The user agent should be set once per session as a single string of the form \texttt{"Name email"}:
\begin{lstlisting}
julia> using EDGAR
julia> set_user_agent("Your Name name@example.com")
\end{lstlisting}

\noindent If the \texttt{User-Agent} header is not set, \texttt{EDGAR.jl} will raise an error locally and stop execution before making any request to the SEC, ensuring compliance with the SEC's access policy.

\subsection{Persisting an User-Agent}
The function \texttt{set\_user\_agent()} sets the User-Agent only for the \emph{current} Julia process. As a result, a new REPL session, script execution, or notebook kernel will start without this configuration. The resolution order used by \texttt{EDGAR.jl} is:
\begin{enumerate}[nosep]
  \item an explicit value set via \texttt{set\_user\_agent()}; otherwise
  \item the \texttt{SEC\_USER\_AGENT} environment variable; otherwise
  \item an error is thrown.
\end{enumerate}



\noindent To avoid configuring the User-Agent in every session, it can be defined once as an environment variable. This variable is automatically read by all future Julia processes, including REPL sessions, scripts, and notebook kernels. It can be set in \texttt{startup.jl} or exported from the shell,
\begin{lstlisting}
ENV["SEC_USER_AGENT"] = "Your Name name@example.com"
\end{lstlisting}

The User-Agent is not a secret: it is simply the identity transmitted to the SEC with each request. For this reason, storing it in a startup file is safe and recommended.

Alternatively, \texttt{EDGAR.jl} provides convenience functions to manage persistence automatically,
\begin{lstlisting}
julia> persist_user_agent("Your Name name@example.com")
julia> unpersist_user_agent()   # removes the persisted setting
\end{lstlisting}

\section{Quick Start}
\label{sec:quickstart}

\subsection{Searching}
\label{sec:qs-search}

Resolve a ticker to a CIK, then search the text of filings:

\begin{lstlisting}
using EDGAR
set_user_agent("Jane Doe jane@example.com")

# Ticker -> 10-digit, zero-padded CIK (exact match -> 0 or 1 row)
id = only(cik("AAPL"; by = :ticker)).cik   # "0000320193"

# Full-text search across filing contents (2001 onward), restricted to 10-K forms
rows = filings_by_text("climate risk"; forms = "10-K")   # a Tables.jl row table
for f in first(rows, 5)
    println(f.filed, "  ", f.form, "  ", f.entity)
end
\end{lstlisting}


\subsection{Filing Formats: Documents and iXBRL}
\label{sec:qs-reports}

\emph{Structured extraction of financial statements from inline XBRL (iXBRL) is
planned and not yet available.} The SEC tags the financial statements directly
inside each filing's primary HTML document; a future release will parse those
\texttt{<ix:\dots>} tags into tidy tables.

In the meantime you can already work with the filing \emph{documents}: list a
filer's filings, download a filing, convert it to text, and pull named sections
(see \S\ref{sec:api-filings}). For the \emph{tagged numbers} that underlie the
statements today, use the XBRL data API above (\S\ref{sec:qs-findata}) --- the
SEC has already extracted those facts for you, so you rarely need to parse the
iXBRL by hand.


\subsection{Financial Data}
\label{sec:qs-findata}

The three XBRL endpoints share the \texttt{us-gaap} vocabulary and return JSON:

\begin{lstlisting}
# Every XBRL fact a company has reported, in one document
facts = company_facts("320193")
gaap  = facts.facts[Symbol("us-gaap")]      # us-gaap has a hyphen: index by Symbol

# One concept over time (net income, grouped by unit)
ni = company_concept("320193", "us-gaap", "NetIncomeLoss")
println(ni.units.USD[end].val)

# The same concept across every filer for one period (a cross-section)
assets = xbrl_frames("us-gaap", "Assets", "USD", "CY2022Q4I")
println(length(assets.data), " filers reported Assets")
\end{lstlisting}

\noindent
A trailing \texttt{I} on the frame period (\texttt{"CY2022Q4I"}) denotes an
\emph{instant} (a balance-sheet point in time); drop it for a \emph{duration}
(a flow such as revenue). See the Glossary (\S\ref{sec:glossary}).


\section{Form Types and Entities}

For most corporate analysis, the “big three” report forms (10-K, 10-Q, and 8-K) provide the majority of the useful information. However, several other SEC forms can be extremely valuable depending on what we are researching and analysing.

\medskip
\noindent Using \texttt{EDGAR.jl} to retrieve any form, first pass its EDGAR code\footnote{The code is the name used in each heading \$\ref{big3}-\$\ref{hii} below, except where noted (e.g.\ \texttt{"SC 13D"}, \texttt{"13F-HR"}, \texttt{"NPORT-P"}, \texttt{"N-MFP2"}).} (e.g annual report "10-K", quarterly report "10-Q", current report "8-K") to,
\begin{itemize}
  \item \texttt{filings\_by\_cik}
    \begin{lstlisting}
    filings_by_cik(320193; forms = "8-K")          # Apple CIK = 320193
    \end{lstlisting}
  \item \texttt{filings\_by\_text}
    \begin{lstlisting}
    filings_by_text("supply chain disruption"; forms = ["10-K", "10-Q"]) 
    \end{lstlisting}
\end{itemize}

\subsection{“Big Three” Report Forms}\label{big3}

\subsubsection{10-K: Annual Report}
The annual report, denoted 10-K, provides a comprehensive overview of a company's business and financial performance for the fiscal year.

\subsubsection{10-Q: Quarterly Report}
			
The quarterly report, denoted 10-Q, updates investors on the company's performance during a fiscal quarter. There is typically no Q4 10-Q because the annual 10-K covers the full year. Unlike a 10-K, the financial statements are usually reviewed but not fully audited.

\subsubsection{8-K: Current Report}

The current report, denoted 8-K, is filed whenever a material event occurs that investors should know about. Unlike 10-K and 10-Q filings, 8-K filings are event-driven and can be filed at any time. Common triggers include:
\begin{itemize}
  \item Acquisitions or mergers
  \item Bankruptcy
  \item CEO or CFO changes
  \item Director appointments or resignations
  \item Major contracts
  \item Earnings announcements
  \item Changes in auditors
  \item Delisting notices
\end{itemize}


\subsection{Foreign Companies}

\subsubsection{20-F: Annual Report}
The foreign-issuer equivalent of Form 10-K. Filed annually by foreign companies whose securities are traded in U.S. markets. Useful for understanding a foreign company’s business, risks, financial condition, and corporate governance. Examples include Toyota Motor Corporation and SAP SE.

\subsection{Corporate Governance}
\subsubsection{Proxy Statement (DEF 14A)}
Many researchers consider the Proxy Statement (DEF 14A)\footnote{\textbf{DEF} denotes Definitive and \textbf{14A} refers to proxy materials filed under Section 14(a) of the Securities Exchange Act.} the best source of information on executive compensation, directors, and corporate governance. Typical contents include:
\begin{itemize}
\item Executive compensation
\item Director biographies
\item Board composition
\item Related-party transactions
\item Shareholder proposals
\end{itemize}


\subsection{Tender Offers and Mergers}
\subsubsection{Schedule TO}
Tender offer filing used in acquisitions, buyouts, and going-private transactions. Useful for understanding the terms, financing, and parties involved in a tender offer.
\begin{itemize}
\item Acquisitions
\item Buyouts
\item Going-private transactions
\end{itemize}



\subsection{Securities Offerings}

\subsubsection{Form S-1}
Form S-1 is the registration statement typically used by companies seeking to go public through an initial public offering (IPO). It is useful for understanding a company before its IPO. Typical contents include:
\begin{itemize}
  \item Detailed business description
  \item Risk factors
  \item Use of proceeds
  \item Management team
  \item Industry overview
\end{itemize}

\subsubsection{Form S-3}
Form S-3 is a simplified registration statement used by eligible seasoned issuers to register additional securities offerings. It is commonly used for follow-on offerings and shelf registrations.


\subsection{Ownership and Insider Activity}

\subsubsection{Schedule 13D}
Filed when an investor acquires more than 5\% of a company's shares and intends to influence management or corporate policies. Useful for identifying activist investors, takeover attempts, and significant ownership positions.

\subsubsection{Schedule 13G}
A simplified alternative to Schedule 13D for passive investors who acquire more than 5\% of a company's shares without seeking to influence management or control of the company.

\subsubsection{Form 4}
Reports purchases and sales of company securities by directors, officers, and beneficial owners holding more than 10\% of the company's shares. Useful for tracking insider trading activity.

\subsection{Holdings of Institutional Investors}\label{hii}

\subsubsection{Form 13F}
An aggregated portfolio disclosure by an institutional investment manager. It primarily reports the holdings managed by the filing institutional investment manager, rather than a detailed fund-by-fund portfolio breakdown. The EDGAR form code is \texttt{"13F-HR"} (the holdings report; \texttt{"13F-NT"} is a notice that another manager reports the holdings).

\subsubsection{Form N-PORT}
Individual registered fund portfolio disclosure, providing monthly portfolio reports required for registered open-end investment companies, including most mutual funds and ETFs \footnote{Main exemptions/exceptions include: (1) money market funds, which are primarily regulated under Form N-MFP instead (\S\ref{N-MFP}); (2) exchange-traded products that are not 1940 Act funds; (3) funds relying on master-feeder or umbrella structures; (4) newly formed or liquidating funds may benefit from transitional relief and reduced or modified reporting, respectively; (5) small fund conditional exemptions (limited scope).}. Each filing corresponds to a single registered investment fund. N-PORT filings are typically made public with a delay (generally $\sim$60 days after month-end), and therefore are not real-time.

The public EDGAR form code is \texttt{"NPORT-P"}. N-PORT is highly granular compared to Form 13F:
\begin{itemize}
  \item Individual securities are reported
  \item CUSIPs\footnote{A CUSIP (Committee on Uniform Security Identification Procedures) is a unique nine-character code used to identify North American financial securities, such as stocks, bonds, and mutual funds. Functioning much like a serial number, it uniquely identifies a security and facilitates the accurate tracking of investments, trade processing, and settlement by brokerages, banks, and clearing organizations.} are included
  \item Positions are disclosed at the security level\footnote{Certain holdings may receive confidential treatment in limited, regulator-approved cases.}
  \item Valuation per holding is provided
\end{itemize}

\noindent N-PORT filings may also include portfolio-level classifications and summaries, including,
\begin{itemize}
  \item Equity vs fixed income vs derivatives breakdowns
  \item Sector and asset-type classifications
  \item Country and currency exposure summaries
  \item Liquidity categories
\end{itemize}

\noindent N-PORT filings may further include risk and exposure metrics (where applicable), including,
\begin{itemize}
  \item Portfolio duration (for fixed income)
  \item VaR-style risk measures (where applicable)
  \item Concentration levels
  \item Exposure breakdowns across risk factors
\end{itemize}

\subsubsection{Form N-MFP}\label{N-MFP}
Most money market funds are not required to file N-PORT in the same way as other mutual funds. They are primarily regulated under Form N-MFP instead, which is tailored specifically to money market instruments. The current EDGAR form code is \texttt{"N-MFP2"} (\texttt{"N-MFP3"} from the 2024 amendments).

\subsection{Entities}

Every filer on EDGAR, whoever submits a filing, is an \emph{entity},
identified by a \textbf{CIK} (Central Index Key, \S\ref{sec:glossary}). An entity is
not always an operating company: it may be a registered \emph{investment} company (a
mutual fund, ETF, or money-market fund), an institutional investment manager, or an
individual (e.g.\ a director filing Form 4). The SEC tags each filer with an
\texttt{entityType}, and which forms it files follows from that type:

\begin{itemize}
  \item \texttt{"operating"} --- an operating company (or an institutional manager).
    Files the corporate forms above: 10-K / 10-Q / 8-K, DEF 14A, S-1/S-3, 20-F for
    foreign issuers; a manager also files \textbf{13F}.
  \item \texttt{"investment"} --- a registered investment company: a mutual fund or
    ETF (files \textbf{N-PORT}), or a money-market fund (files \textbf{N-MFP}).
\end{itemize}

\noindent \texttt{EDGAR.jl} exposes entities through three functions. \texttt{cik} maps a name
or ticker to a CIK, covering entities that \emph{have} a ticker (companies and
listed ETFs/funds). \texttt{profile(cik)} returns a single filer's row-invariant
record --- its \texttt{name}, \texttt{entityType}, SIC industry, fiscal year-end,
tickers, former names, and so on. And the \texttt{entity} column of
\texttt{filings\_by\_text} is simply that filer's name. So a typical workflow is:
resolve the CIK (\texttt{cik}), inspect the filer (\texttt{profile}), then pull its
filings of the relevant form (\texttt{filings\_by\_cik}).

\begin{lstlisting}
julia> id = only(cik("AAPL"; by = :ticker)).cik
# Output:
"0000320193"

julia> profile(id).entityType 
"operating"

reports = filings_by_cik(id; forms = "10-K")
table(reports)

\end{lstlisting}

\section{API}
\label{sec:api}

CIKs are the SEC Central Index Key, zero-padded to 10 digits (Apple $\to$
\texttt{0000320193}). Any \texttt{cik} argument accepts an integer or a string, with
or without leading zeros (\texttt{320193}, \texttt{"320193"} and \texttt{"0000320193"}
are equivalent); functions normalize to the 10-digit form. Every data call goes
through the cached, User-Agent-aware request layer (\S\ref{sec:caching}).

\subsection{Searching and Entity Lookup}
\label{sec:api-search}

Searching typically begins with identifying a filer and then retrieving either company metadata or relevant filings. The package therefore provides functions for company lookup, filer profiling, full-text search of filing contents, and retrieval of filings submitted by a specific entity. Table~\ref{tab:searching} summarizes these functions and their intended usage.

\subsubsection{\texttt{cik}}

\paragraph{Signatures}
\begin{description}
  \item[\sig{cik()}] 
  \item[\sig{cik(entity; by = :name | :ticker | :any)}] 
\end{description}



\paragraph{Selection Criteria}
\begin{itemize}
  \item \texttt{cik()} returns all entities with an associated CIK code in the EDGAR dataset.
  \item \texttt{cik(entity\_query; by = :name | :ticker | :any)} selects entities by matching the input string against the company name, ticker, or both, depending on the \texttt{by} argument.
\end{itemize}
\paragraph{Keyword Arguments}
\begin{description}[nosep]

  \item[\quad - \texttt{by}]~
  \begin{description}[nosep]
    \item[\quad \texttt{:any}] (default) matches rows where \texttt{entity\_query} appears either as a substring in the entity name (case-insensitive) or as an exact ticker match. For example, \texttt{"IBM"} matches the ticker IBM, while \texttt{"apple"} matches company names containing that string. Each row is tested once, so a row matching both appears only once.
    \item[\quad \texttt{:name}] returns rows where the entity name contains \texttt{entity\_query} (case-insensitive substring match).
    \item[\quad \texttt{:ticker}] returns at most one row matching the exact ticker (case-insensitive).
  \end{description}
\end{description}

\paragraph{Return Value} Both forms always return a \texttt{Tables.jl} \emph{row table}, with fields 
\begin{itemize}
  \item \texttt{entity}
  \item \texttt{ticker}
  \item \texttt{cik}
\end{itemize}
with one record per ticker.

The return type is consistent across all modes, ensuring type stability. The result behaves as a standard Tables.jl-compatible object and can be directly consumed by downstream tools such as \texttt{DataFrames.jl}, \texttt{TimeSeries.jl}, \texttt{CSV.jl}, \texttt{XLSX.jl}, \texttt{Parquet.jl}, or database interfaces such as \texttt{SQLite.jl}.

A query may yield several rows, because (a) one company can have multiple tickers
(e.g. share classes \texttt{GOOGL}/\texttt{GOOG}), and (b) several companies may match
the query in their name.

\paragraph{Julia Examples:} Finding the CIK code for an entity.

\begin{lstlisting}
julia> cik("alphabet")           # by = :any (default); "alphabet" matches the name
# Output:

\subsubsection{\texttt{full\_text\_search}}

\paragraph{Signature}
\begin{description}
  \item[\sig{full\_text\_search(text; exact=true, forms=nothing, startdate=nothing, enddate=nothing, from=0)}]
\end{description}

\noindent \texttt{full\_text\_search} is the SEC's own name for this endpoint (EFTS); it is also exported under the alias \texttt{filings\_by\_text}, and pairs with \texttt{filings\_by\_cik}. The function
\texttt{full\_text\_search} discovers \textit{filings} by their content (e.g. "which filings mention 'supply chain disruption'?"). Search the full text of SEC filings, everything filed from 2001 onward, for some \texttt{text} through the EDGAR full-text search (EFTS) API. 

\paragraph{Keyword Arguments}
\begin{description}[nosep]
  \item[\quad - \texttt{exact}]
  \item[\quad - \texttt{forms}] may be a single string (\texttt{"10-K"})\footnote{Form 10-K is a detailed annual filing submitted by public companies to the SEC. It provides a comprehensive overview of the company's business activities, financial health, and key risks. The filing includes audited financial statements, management's discussion of financial results, information about the company's operations and strategy, disclosures on legal and regulatory issues, and an assessment of risk factors that could affect future performance.} or a collection; 
  \item[\quad - \texttt{startdate}] is a \texttt{"YYYY-MM-DD"} format string
  \item[\quad - \texttt{enddate}] is a \texttt{"YYYY-MM-DD"} format string 
  \item[\quad - \texttt{from}] is the 0-based offset used to page (EFTS returns a fixed page of 100 results per request). 
\end{description}
By default the (\texttt{text}) is matched \textbf{exactly}, so
\texttt{"supply chain disruption"} finds only filings with those three words adjacent;
pass \texttt{exact = false} to match the words loosely instead. Results come back
ranked by relevance, not by date.

\paragraph{Return Values.} The method \texttt{full\_text\_search()} returns a \texttt{Tables.jl} \emph{row table}, a row per matching filing for the current page (up to 100). There is no total-match count; to page, advance (\texttt{from}) in steps of 100 until you get fewer than 100 rows. The returned columns are described in Table~\ref{tab:full_text_search}.

\subsubsection{\texttt{filings\_by\_cik}}

\paragraph{Signature}
\begin{center}
\textbf{\sig{filings\_by\_cik(cik; forms = nothing, startdate = nothing, enddate = nothing, from = 0)}}
\end{center}


\noindent List a single filer's filings (2001 onward) through the same EFTS
endpoint, but keyed by its \texttt{cik} (an integer or string, padded or not) using
EDGAR's \emph{entity} filter --- the web form's company/CIK field. This is a true
``filed by this filer'' query, unlike full-text searching for the CIK as document
text, which also matches filings that merely \emph{mention} it (e.g.\ a fund listing
the company among its holdings). 

\paragraph{Keyword Arguments}
\begin{description}[nosep]
  \item[\quad - \texttt{forms}] behaves as for \texttt{full\_text\_search}
  \item[\quad - \texttt{startdate}] is a \texttt{"YYYY-MM-DD"} format string
  \item[\quad - \texttt{enddate}] is a \texttt{"YYYY-MM-DD"} format string 
  \item[\quad - \texttt{from}] is the 0-based offset used to page (EFTS returns a fixed page of 100 results per request). 
\end{description}

\subsubsection{The \texttt{Filing} Type}
\label{sec:filing-type}

\noindent A \texttt{Filing} is an immutable value representing a filing document in
memory. It is defined as a simple \texttt{struct} with six fields:

\begin{lstlisting}
struct Filing
    cik::String         # 10-digit, zero-padded Central Index Key
    accession::String   # dashed accession number
    document::String    # the primary document's filename
    url::String         # the EDGAR Archives URL it was fetched from
    kind::Symbol        # :ixbrl, :xbrl, or :html
    content::String     # the raw document text (HTML or XML)
end
\end{lstlisting}

\noindent In normal use, \texttt{Filing} instances are created by
\texttt{fetch\_filing}, which populates all fields from the SEC submissions data and
the retrieved document. Because \texttt{Filing} is a plain \texttt{struct}, the
default constructor
\texttt{Filing(cik, accession, document, url, kind, content)} is also available,
which can be useful in tests or when reconstructing a filing from previously saved
content and metadata.

Fields are accessed directly using dot notation. For example,
\texttt{f.content} contains the document text processed by the extraction helpers, \texttt{f.kind} distinguishes between iXBRL, XBRL, and HTML filings, and \texttt{f.url} records the document's source. A \texttt{Filing} also displays as a
compact one-line summary in the REPL and renders inline as HTML in notebook front-ends (\S\ref{sec:display-filing}).


\subsubsection{Displaying a \texttt{Filing}}
\label{sec:display-filing}

\paragraph{Web Browser}
The primary way to inspect a \texttt{Filing} is in a web browser via
\texttt{open\_filing}. The function writes the filing to a temporary HTML file,
downloads any referenced assets (such as images), rewrites the corresponding URLs
to point at the local copies, and then opens the result in the system's default
browser.

\begin{lstlisting}
julia> f = fetch_filing(320193, "0000320193-25-000079")
julia> open_filing(f)
\end{lstlisting}

\noindent For \texttt{:ixbrl} and \texttt{:html} filings, the browser displays the
document exactly as a normal web page. Since iXBRL is itself an HTML document with
embedded XBRL markup, the filing renders as the familiar report submitted to the
SEC, including tables, styling, and images.

A \texttt{:xbrl} filing is different: it contains XML rather than HTML and
therefore has no visual presentation of its own. In that case,
\texttt{open\_filing} wraps the XML in a minimal HTML page and displays it as escaped source code, allowing the underlying instance document to be inspected in the
browser.

\paragraph{Notebook} Besides opening a filing in a browser, a \texttt{Filing} can render
\emph{inline} in any rich display front-end, such as Jupyter or Pluto notebooks,
via its \texttt{text/html} \texttt{show} method. There is nothing to call explicitly:
when a \texttt{Filing} is the last expression in a cell, the front-end requests its
HTML representation and renders the filing directly in the output area.

\begin{lstlisting}
julia> f = fetch_filing(320193, "0000320193-25-000079")
julia> f          # a notebook renders the filing itself, inline
\end{lstlisting}

\noindent Notebooks render the HTML in-page, where relative image references cannot
be resolved against files on disk. To ensure that figures still appear, the
\texttt{show} method replaces each referenced image with a base64-encoded
\texttt{data:} URI — the in-memory analogue of what \texttt{download\_assets} does on
disk — making the rendered filing completely self-contained.

\paragraph{Terminal REPL}
In a plain terminal REPL there is no HTML renderer, so evaluating \texttt{f} displays a compact summary such as
\texttt{Filing(:ixbrl, “aapl-20250927.htm”, 1520208 bytes)}.
Use \texttt{open\_filing} to view the filing in a web browser instead.

\noindent Note that \texttt{:xbrl} filings contain XML rather than HTML. Because there is no visual document to render, they are displayed as escaped source within a \texttt{<pre>} block. Only \texttt{:ixbrl} and \texttt{:html} filings render as formatted documents.
\paragraph{HTML as a String}
To obtain the rendered markup as a \texttt{String} (for embedding or
post-processing), or to trigger rendering explicitly when a \texttt{Filing} is not
the final expression in a function, loop, or notebook cell, use Julia's standard
display API,

\begin{lstlisting}
julia> html = repr("text/html", f)        # the rendered HTML as a String
julia> display(f)                         # render to the active display on demand
\end{lstlisting}

\subsubsection{\texttt{fetch\_filing}}

\paragraph{Signature}
\begin{center}
\textbf{\sig{fetch\_filing(cik, accession; kind = (:auto) | :ixbr | :xbr | :html)}}
\end{center}

\noindent Fetch a single filing's document from the EDGAR \texttt{Archives} into
memory (no disk write). The fetch goes through \texttt{fetch\_url}, so it is
cached and uses the configured User-Agent. 

\paragraph{Keyword Arguments}
\begin{description}[nosep]
  \item[\quad - \texttt{cik}]
  \item[\quad - \texttt{accession}]
  \item[\quad - \texttt{kind}]~
  \begin{description}[nosep]
    \item[\quad \texttt{:auto}] (default) takes the iXBRL primary document when the filing has one (\texttt{isInlineXBRL}), else the classic XBRL instance (\texttt{isXBRL}), otherwise the plain primary HTML.
    \item[\quad \texttt{:ixbrl}] force the primary document
    \item[\quad \texttt{:xbrl}] force the XBRL instance (located via the filing's \texttt{index.json})
    \item[\quad \texttt{:html}] force the primary document
  \end{description}
\end{description}

\paragraph{Return Value.}
A \texttt{Filing} (\S\ref{sec:filing-type}), whose fields are
\begin{itemize}
  \item \texttt{cik}
  \item \texttt{accession}
  \item \texttt{document}
  \item \texttt{url}
  \item \texttt{kind}
  \item \texttt{content} --- the raw document text
\end{itemize}


\paragraph{Julia Example:} Fetch one filing using its accession number and show the primary document's name and format. 

\begin{lstlisting}
julia> f = fetch_filing(320193, "0000320193-25-000079")
# Output:
Filing(:ixbrl, "aapl-20250927.htm", 1520208 bytes)

julia> f.kind, f.document, f.url
# Output:
(:ixbrl, "aapl-20250927.htm", "https://www.sec.gov/Archives/edgar/data/320193/000032019325000079/aapl-20250927.htm")
\end{lstlisting}


\subsubsection{\texttt{open\_filing}}

\begin{center}
\textbf{\sig{open\_filing(f::Filing; assets = true)}}
\end{center}
\begin{center}
\textbf{\sig{open\_filing(cik, accession; kind = :auto, assets = true)}}
\end{center}
\begin{center}
\textbf{\sig{open\_filing(path::AbstractString)}}
\end{center}



\noindent View a fetched \texttt{Filing} in your default browser, returning the path
opened. A browser can only open a file or URL --- never an in-memory string --- so
this writes \texttt{f} to a fresh \emph{temporary} directory (under the filename
\texttt{f.document}, so the extension and tab title are right) and opens it. It is a
throwaway view; use \texttt{save\_filing} to keep a copy. For programmatic use no file
is needed at all. Methods such as \texttt{extract\_section} work straight off
\texttt{f.content}.

Filing's images are stored as separate files \emph{beside} the document in the
EDGAR Archives, and \texttt{fetch\_filing} downloads only the primary document, so the
HTML's relative \texttt{<img>} references would otherwise resolve to nothing and the
figures appear blank. With \texttt{assets = true} (the default) the referenced images and any CSS/JS are fetched into the same temporary directory via
\texttt{download\_assets} so the page renders complete; pass \texttt{assets = false} to
open just the HTML (faster, no extra requests). The temporary directory --- the
document and its images together --- is registered for automatic cleanup when the
Julia process exits, so nothing lingers on disk between sessions.

\begin{lstlisting}
julia> f = fetch_filing(320193, "0000320193-25-000079")
# Output:
Filing(:ixbrl, "aapl-20250927.htm", 1520208 bytes)

julia> open_filing(f)          # renders the 10-K, with its images, in your browser
\end{lstlisting}

\paragraph{Fetch and Open} For the common case where the filing is only being viewed, the convenience method
\texttt{open\_filing(cik, accession; kind = :auto, assets = true)} combines \texttt{fetch\_filing} and \texttt{open\_filing} into a single call, fetching the
filing and opening it immediately in the browser without requiring an intermediate
\texttt{Filing} object:

\begin{lstlisting}
julia> open_filing(320193, "0000320193-25-000079")   # fetch and view in one step
\end{lstlisting}

\paragraph{Reopen from Disk} A third method, \texttt{open\_filing(path::AbstractString)},
opens a filing that is \emph{already saved on disk} --- the on-disk counterpart of the
in-memory \texttt{Filing} method. It is the natural partner of \texttt{save\_filing}
(which returns the path it wrote) and works for any saved HTML page, including an
extracted section such as a balance sheet (still part of the filing). The path must
exist; an \texttt{http(s)://} URL is passed through to the browser as-is.

\begin{lstlisting}
julia> path = save_filing(f; destdir = "filings")
julia> open_filing(path)          # reload from disk and view in the browser
\end{lstlisting}


\subsubsection{\texttt{extract\_section}}

\begin{center}
\textbf{\sig{extract\_section(html, names; base\_path = nothing)}}
\end{center}


\noindent Pull named sections from filing HTML/text by heuristic,
case-insensitive matching --- e.g.\ \texttt{"Item 7"} or
\texttt{"Management's Discussion"}. Returns a \texttt{Dict} mapping each requested name
to its extracted text (the full section, up to where the next section begins).


\subsubsection{\texttt{save\_filing}}

\begin{center}
\textbf{\sig{save\_filing(f::Filing; destdir = ".", assets = true)}}
\end{center}
\noindent Write a fetched \texttt{Filing} verbatim to \texttt{destdir/f.document}, creating \texttt{destdir} if needed, and return the path. With
\texttt{assets = true} (the default) the filing's images and any CSS/JS are also
downloaded into \texttt{destdir} via \texttt{download\_assets}, so the saved copy renders
self-contained. Pass \texttt{assets = false} to write only the document. 


\subsubsection{\texttt{download\_assets}}

\begin{center}
  \textbf{\sig{download\_assets(f::Filing; destdir = ".")}}
\end{center}


\noindent Download the relative assets --- chiefly the embedded images --- that a
filing's HTML references, writing each next to the document under \texttt{destdir} so
a saved filing renders self-contained. The \texttt{src}/\texttt{href} attributes are
scanned and only relative URLs with a known asset extension (images, CSS, JS) are
taken, so links to sibling filings and in-page anchors are skipped; each asset is
fetched through the cached \texttt{fetch\_url} and written preserving any sub-path. The
download is best-effort (a reference that cannot be fetched is skipped) and the list
of filenames written is returned. \texttt{open\_filing} and \texttt{save\_filing} call it
for you; use it directly only to populate a directory yourself.


\subsubsection{Worked Example: The Filings Life-Cycle}
\label{sec:lifecycle-example}

\noindent The functions above compose into a complete workflow. The script below
walks the whole life-cycle on a real filing --- Walmart's latest Form 10-Q (chosen
over a tidier filer precisely because its Balance Sheet extracts a little roughly,
exercising the heuristics) --- using nothing but the public API: it resolves the
CIK, fetches the 10-Q, renders it, extracts the Balance Sheet, saves both the full
filing and the Balance Sheet to disk, and finally reloads each from disk and shows
it in the browser.

\begin{lstlisting}
julia> using EDGAR
julia> set_user_agent("Your Name you@example.com")   

# 1. Find the company's CIK from its ticker.
julia> row = only(cik("WMT"; by = :ticker))

# 2. Fetch its latest Form 10-Q into memory.
julia> filing = first(filings_by_cik(row.cik; forms = "10-Q"))
julia> f = fetch_filing(row.cik, filing.accession)

# 3. Render the fetched filing in the browser.
julia> open_filing(f)

# 4. Extract the Balance Sheet from the filing HTML (f.content).
julia> balance_sheet = extract_section(f.content, ["Balance Sheet"])["Balance Sheet"]

# 5. Save the full 10-Q to disk (also downloads its images).
julia> path_10q = save_filing(f; destdir = "filings_demo")

# 6. Save the extracted Balance Sheet as a small standalone HTML page.
julia> path_bs = joinpath("filings_demo", "balance_sheet.html")
julia> write(path_bs, "<!doctype html><html><body><pre>" *
               balance_sheet * "</pre></body></html>")

# 7. Load the saved 10-Q from disk and show it in the browser.
julia> open_filing(path_10q)

# 8. Load the saved Balance Sheet from disk and show it in the browser.
julia> open_filing(path_bs)
\end{lstlisting}

\noindent Every step succeeds against the live filing; Table~\ref{tab:lifecycle}
records what each produces (the public API only, no internals).

\begin{table}[h]
\centering

\subsubsection{Interactive Extraction: \texttt{select\_section}}
\label{sec:select-section}

\sig{select\_section(f::Filing; timeout=1800) -> Selection}

\noindent Sometimes the region you want is easier to \emph{point at} than to name.
\texttt{select\_section} opens the rendered filing \texttt{f} in your browser with an
interactive picker and hands the region you choose back to Julia as a
\texttt{Selection}. A local HTTP server --- bound to \texttt{127.0.0.1} on a random
port, so nothing is exposed off the machine --- serves the document with a small
picker script injected, receives your pick, and shuts down as soon as one arrives or
\texttt{timeout} seconds elapse. (Pass a larger \texttt{timeout} if you expect to
browse for a while before choosing.)

\paragraph{Hover, widen, capture.} As you move the cursor the picker outlines the
element beneath it and a badge names what will be captured. A table is completely
covered by its own cells, so you can never hover the \htmltag{table} directly --- the
cursor always lands on a \htmltag{td}. Press \textbf{$\uparrow$} to widen the
selection to the parent element (and \textbf{$\downarrow$} to narrow); the highlight
\emph{sticks} as you press, so a stray mouse movement does not snap it back. Press
\textbf{Enter} (or click) to capture, or \textbf{Esc} to cancel.

\paragraph{The colour tells you what you will get.} The outline and badge are
colour-coded by the \emph{kind} of the candidate element, which is exactly what
determines how the capture is later exported. Widening from a number cell in a
financial statement cycles through:

\begin{center}\ttfamily
\htmltag{td}~\swatch{pickerpurple} $\to$ \htmltag{tr}~\swatch{pickerpurple} $\to$
\htmltag{tbody}~\swatch{pickerpurple} $\to$ \htmltag{table}~\swatch{pickergreen} $\to$
\htmltag{div}~\swatch{pickeramber} (wrapper)
\end{center}

\begin{table}[h]
\centering
\caption{What each highlight captures. The same block on screen, but a different
payload and a different \texttt{kind}.}
\label{tab:picker-kinds}
\begin{tabular}{@{}p{0.21\textwidth}p{0.10\textwidth}p{0.34\textwidth}p{0.25\textwidth}@{}}
\toprule
\textbf{Highlight} & \textbf{\texttt{kind}} & \textbf{What is captured} & \textbf{Good for} \\
\midrule
\swatch{pickergreen}~\htmltag{table} & \texttt{:table} &
  The clean \htmltag{table} --- header and body rows, with every XBRL fact inside. &
  The right pick: a structured Markdown table and resolved facts. \\
\swatch{pickeramber}~\htmltag{div} (wrapper) & \texttt{:mixed} &
  The table \emph{plus} the prose in its wrapper --- the
  ``(Amounts in millions\ldots)'' caption, footnotes. &
  A table together with its caption and notes (context for RAG). \\
\swatch{pickerpurple}~\htmltag{tbody} & \texttt{:prose} &
  The body rows only, as text --- \emph{not} parsed as a table, and it can omit the
  column headers that sit above \htmltag{tbody}. &
  Rarely what you want for a table. \\
\swatch{pickerpurple}~\htmltag{tr} / \htmltag{td} & \texttt{:prose} &
  One row / one cell, as text. & A single line or figure. \\
\bottomrule
\end{tabular}
\end{table}

\noindent So even though \htmltag{tbody} (purple) and \htmltag{table} (green) look
like the same block, the green one round-trips as a structured table --- with its
headers and facts --- while the purple \htmltag{tbody} comes back as a flat blob of
text with no columns. \emph{That} is why the colour coding matters: it tells you what
you will actually get, not just what is outlined.

\paragraph{Prose is a feature, not a mistake.} A purple (\texttt{:prose}) pick is not
a failed table pick --- it is its own use case. The narrative parts of a filing are
often exactly what you are after: Management's Discussion and Analysis, a risk factor,
a single footnote, the cover page. The picker captures those as readily as a table;
reach for green when you want a grid of numbers, and purple when you want the words.

\paragraph{Confirmation --- visual, audible, and in Julia.} On a successful capture the
toolbar reports what was taken (e.g.\ ``Captured TABLE --- 23 rows $\times$ 3 cols'', or
``\ldots\ 1{,}214 chars'' for prose), and a short synthesised ``shutter'' click plays ---
generated with the browser's Web Audio API, so no sound file is shipped. Back in Julia,
\texttt{select\_section}/\texttt{select\_sections} log a one-line \texttt{@info} summary on
return (kind, selector, size), so you can see the pick landed without inspecting the value.

\paragraph{Capturing several regions: \texttt{select\_sections}.} To grab more than one
region in a single sitting --- the income statement, the balance sheet, a footnote --- use
the plural form:

\sig{select\_sections(f::Filing; timeout=1800) -> Vector\{Selection\}}

\noindent It behaves like \texttt{select\_section}, except each \textbf{Enter}/click
\emph{adds} the highlighted region (the toolbar keeps a running count) and \textbf{Esc}
\emph{finishes}, returning everything as a \texttt{Vector\{Selection\}}. Repeated picks of
the same element are de-duplicated by selector, and finishing without picking anything
returns an empty vector (it does not throw, unlike the singular form). The two functions
keep distinct, fixed return types --- \texttt{Selection} versus \texttt{Vector\{Selection\}}
--- rather than overloading one with a mode flag. Note the difference in \textbf{Esc}: in
\texttt{select\_section} it \emph{cancels} (nothing was chosen, so the call throws); in
\texttt{select\_sections} it \emph{finishes} (returns what you have so far).

\paragraph{Previewing a pick: \texttt{open\_filing(sel)}.} A \texttt{Selection} is not a
\texttt{Filing}, so the \texttt{Filing} verbs do not apply to it --- \texttt{save\_filing(sel)}
is a \texttt{MethodError}. There is, however, a \texttt{Selection} method of
\texttt{open\_filing} for a quick visual check:

\sig{open\_filing(sel::Selection) -> String}

\noindent It wraps the captured HTML in a minimal page --- a \htmltag{base} so the
fragment's relative images still resolve to the SEC Archives, plus a provenance header
(filer, accession, selector) --- writes it to a throwaway temporary file and opens it,
returning the path. To \emph{keep} a selection (as Markdown, facts, or a database row), use
the export layers rather than \texttt{save\_filing}.

\paragraph{Exporting to Markdown: \texttt{markdown}.} Turn a pick into Markdown --- a
structured table becomes a Markdown table, prose becomes text --- ready to drop into a
document or hand to a language model:

\sig{markdown(sel::Selection; provenance=true) -> String}

\noindent The faithful capture grid (Step 2.1) is cleaned up here. SEC iXBRL tables put
the \texttt{\$} marker in its own column on only some rows (the first of a section,
totals, per-share lines), which shifts the value from one column to another row by row;
\texttt{markdown} drops the empty and lone-\texttt{\$} cells in each row and re-packs, so
the columns line up again, and uses the first surviving row as the header. With
\texttt{provenance = true} (the default) a blockquote recording the source --- CIK,
accession, URL, selector --- is prepended, so the output is self-describing for
citation/RAG (Step 2.3); pass \texttt{provenance = false} for just the table. Walmart's
income statement renders as (abridged):

\begin{verbatim}
> Source: SEC EDGAR -- CIK 0000104169, accession 0000104169-26-000102 (table)
> Selector: `div:nth-of-type(52) > table:nth-of-type(1)`

| Three Months Ended April 30, |  |  |
| --- | --- | --- |
| (Amounts in millions, except per share data) | 2026 | 2025 |
| Net sales | 175,684 | 163,981 |
| Total revenues | 177,751 | 165,609 |
| Interest income | (79) | (93) |
\end{verbatim}

\begin{lstlisting}
f    = fetch_filing(104169, "0000104169-26-000102")  # Walmart 10-Q
sel  = select_section(f)        # hover a number, press up to <table> (green), Enter
sel.kind                        # :table
open_filing(sel)                # preview exactly what you captured
print(markdown(sel))            # Markdown table under a provenance header

sels = select_sections(f)       # pick several regions, then Esc to finish
length(sels)                    # how many you captured
\end{lstlisting}

\paragraph{Extracting XBRL facts: \texttt{facts}.} When the filing is inline XBRL, the
picker also resolves the machine-readable \emph{facts} tagged inside the selection
against the document's \texttt{<ix:header>} (its contexts and units). \texttt{facts}
assembles them into a Tables.jl row table --- the hardened fact schema:

\sig{facts(sel::Selection) -> Vector\{FactRow\}}

\noindent Each row carries \texttt{cik}, \texttt{accession}, \texttt{statement},
\texttt{concept} (e.g.\ \texttt{us-gaap:Revenues}), \texttt{label}, the \emph{normalised}
\texttt{value} (\texttt{displayed} $\times\,10^{\mathrm{scale}}\times$ \texttt{sign}),
\texttt{unit} (\texttt{USD}, \texttt{shares}, \texttt{USD/shares}), \texttt{period\_start}/
\texttt{period\_end}, \texttt{is\_instant} (a balance-sheet instant vs an income-statement
flow), \texttt{dimensions} (JSON), \texttt{decimals}, the raw \texttt{context\_ref}/
\texttt{unit\_ref}, and \texttt{source\_selector}. Rows are de-duplicated on the natural
key \texttt{(accession, concept, context\_ref, unit\_ref)}. A prose-only selection yields
an \emph{empty} table --- no error. As a \texttt{Vector} of \texttt{NamedTuple}s it is a
Tables.jl source, so it renders with \texttt{PrettyTables} and feeds \texttt{CSV}, Arrow,
\texttt{DataFrames} or a database.

\begin{lstlisting}
sel = select_section(f)         # pick the income statement
using PrettyTables
pretty_table(facts(sel))        # the normalised facts as a table
\end{lstlisting}

\noindent\emph{Correctness.} The extraction is validated against the SEC's own
\texttt{companyfacts} API (\S\ref{sec:api-xbrl}): for Walmart's 10-Q, every fact picked
from the income statement (42) and the balance sheet (98) --- durations and instants,
across all units --- equals the API's value for the same concept, period and unit. The
picked numbers are the reported numbers.

\subsubsection{Exporting a Selection: iXBRL, Markdown, Facts JSON, DuckDB}
\label{sec:export-layers}

A picked \texttt{Selection} exports at three levels of abstraction --- \emph{preserve} it,
\emph{read} it, or \emph{analyse} it --- across four formats (Table~\ref{tab:export-layers}).
\texttt{save\_selection} is the one-stop ``Export As'' menu; each format is also available
on its own.

\begin{table}[h]
\centering
\caption{The export layers of a \texttt{Selection}.}
\label{tab:export-layers}
\begin{tabular}{@{}p{0.13\textwidth}p{0.14\textwidth}p{0.30\textwidth}p{0.30\textwidth}@{}}
\toprule
\textbf{Layer} & \textbf{Format} & \textbf{Function} & \textbf{Purpose} \\
\midrule
Presentation & iXBRL HTML & \texttt{sel.html} / \texttt{open\_filing} & Preserve the fragment exactly (lossless) \\
Presentation & Markdown & \texttt{markdown(sel)} & Documentation, notes, RAG \\
Semantic & Facts JSON & \texttt{facts\_json(sel)} & Portable XBRL semantics; LLM input; old-XBRL bridge \\
Analytical & DuckDB & \texttt{to\_duckdb(sel, db)} & A queryable fact warehouse \\
\bottomrule
\end{tabular}
\end{table}

\sig{save\_selection(sel::Selection; as, dir=".", db=nothing) -> String | Int}

\noindent \texttt{as} is \texttt{:ixbrl}, \texttt{:markdown}, \texttt{:facts} or
\texttt{:duckdb}. The first three write a file under \texttt{dir} (named
\texttt{<accession>\_<selector-slug>.<ext>}) and return its path; \texttt{:duckdb} appends
to a warehouse (default \texttt{<dir>/facts.duckdb}) and returns the number of rows inserted.

\begin{lstlisting}
sel = select_section(f)                            # pick the income statement
save_selection(sel; as = :ixbrl,    dir = "out")   # lossless fragment .ixbrl.html
save_selection(sel; as = :markdown, dir = "out")   # .md
save_selection(sel; as = :facts,    dir = "out")   # .facts.json
using DuckDB
save_selection(sel; as = :duckdb,   dir = "out")   # rows appended to out/facts.duckdb
\end{lstlisting}

\paragraph{Facts JSON --- the semantic layer.} \texttt{facts\_json} serialises the resolved
facts to a portable JSON document: a provenance header plus a \texttt{facts} array, values
already normalised. It is the convenient working representation of the iXBRL semantics ---
clean to feed a language model, cheap to re-query, and (carrying no HTML) a \emph{common
shape across modern inline XBRL, old classic XBRL, and extracted fragments}.
\texttt{read\_facts\_json} reads it back into a \texttt{Selection}, so it round-trips and
flows on to \texttt{facts}/\texttt{to\_duckdb} --- which is also how a pre-iXBRL filing,
with no rendered HTML, enters the same pipeline.

\sig{facts\_json(sel::Selection; pretty=true) -> String}
\sig{read\_facts\_json(source) -> Selection}

\begin{verbatim}
{ "version": 1,
  "cik": "0000104169", "accession": "0000104169-26-000102",
  "selector": "div:nth-of-type(52) > table:nth-of-type(1)", "kind": "table",
  "facts": [
    { "concept": "us-gaap:Revenues", "label": "Total revenues",
      "value": 177751000000, "unit": "USD",
      "period_start": "2026-02-01", "period_end": "2026-04-30",
      "is_instant": false, "dimensions": {}, "decimals": -6,
      "context_ref": "c-1", "unit_ref": "usd" } ] }
\end{verbatim}

\paragraph{DuckDB --- the analytical warehouse.} \texttt{to\_duckdb} writes into a DuckDB
warehouse (a package extension --- available after \texttt{using DuckDB}): a single
\texttt{.duckdb} file holding three tables joined by \texttt{accession} ---
\texttt{documents} (the lossless iXBRL HTML, Layer 1, written by
\texttt{to\_duckdb(f::Filing, db)}), \texttt{extractions} (the Facts JSON snapshot of each
pick or filing, Layer 2), and \texttt{facts} (the rows, Layer 3) --- plus an
\texttt{edgar\_meta} schema marker. The \texttt{facts} table is the canonical sink every
source maps into: the picker (\texttt{source='picker'}, with \texttt{source\_selector}
tracing each row back to the picked region) and whole-filing extraction
(\texttt{source='filing'}); the \texttt{form}/\texttt{fy}/\texttt{fp}/\texttt{frame} columns
are reserved for structured-data sources. Its primary key is each fact's \emph{semantic
identity} \texttt{(cik, accession, concept, unit, period, dimensions)} --- not the
document-internal \texttt{context\_ref}/\texttt{unit\_ref}, kept only as provenance --- so
rows are inserted \texttt{ON CONFLICT DO NOTHING}: re-importing a filing is idempotent, and
the same fact from two sources collapses to one row. You grow the warehouse filing by filing,
and every fact joins back to its source document and extraction.

\sig{to\_duckdb(data, db; table="facts") -> Int}

\noindent \texttt{statement\_view} pivots the long fact table back into a \emph{wide}
statement --- one row per concept/label, one column per period --- and, because the warehouse
may hold many filings, \emph{stitches} the statement across them:

\sig{statement\_view(db; statement=nothing, accession=nothing, consolidated=true, months=nothing, by=:concept) -> Vector\{NamedTuple\}}

\noindent \texttt{statement} restricts to one financial statement (e.g.\ \texttt{"IncomeStatement"};
needs facts ingested with \texttt{classify=true}); \texttt{months} is \emph{smart period
selection} --- keep only durations of about that many months (\texttt{3} quarterly, \texttt{12}
annual) plus instants, so periods sharing an end date do not collide; and
\texttt{by=:standard\_concept} groups by the standardized concept (\S\ref{sec:export-layers}) for
cross-company comparison. For example, a quarterly income statement stitched across several
10-Qs:

\begin{lstlisting}
using DuckDB, PrettyTables
archive_filings(104169, "wmt.duckdb"; forms = "10-Q", limit = 4, classify = true)
pretty_table(statement_view("wmt.duckdb"; statement = "IncomeStatement", months = 3,
                            by = :standard_concept))
\end{lstlisting}

\noindent The result is a Tables.jl row table (the period dates are the column names).

\paragraph{Bulk extraction: \texttt{facts(::Filing)} and \texttt{archive\_filings}.} The picker
is for curated, one-at-a-time selection; for whole filings there is a native, browser-free
extractor. \texttt{facts(f::Filing)} reads \emph{every} XBRL fact from a filing --- its inline
XBRL (\texttt{<ix:nonFraction>}) or a classic \texttt{.xml} instance --- as the same row table.
It honours the iXBRL display transforms: a dash shown via \texttt{ixt:fixed-zero} is read as the
value \texttt{0}, and a spelled-out number (\texttt{ixt-sec:numwordsen}, e.g.\ a ``\,No\,'' par
value or ``Forty-two'') is parsed from its words.

\sig{facts(f::Filing; classify=false, labels=false) -> Vector\{FactRow\}}

\noindent and \texttt{archive\_filings} bulk-archives a filer's filings into the warehouse in
one pass: it lists them with \texttt{filings\_by\_cik}, then stores each document and (with
\texttt{facts=true}) its extracted facts (\texttt{source='filing'}) and a filing-level snapshot,
reusing one connection and skipping any filing that fails to fetch.

\sig{archive\_filings(cik, db; forms=nothing, facts=true, classify=false, labels=false, kind=:auto, limit=nothing) -> NamedTuple}

\begin{lstlisting}
using DuckDB
archive_filings(104169, "wmt.duckdb"; forms = "10-Q", limit = 4)
# -> (filings = 4, documents = 4, facts = ...)   # then query across filings
\end{lstlisting}

\paragraph{Statement classification.} The \texttt{statement} column (\texttt{IncomeStatement},
\texttt{BalanceSheet}, \texttt{CashFlow}, \texttt{Equity}, \texttt{ComprehensiveIncome}, the
fund/BDC statements \texttt{ScheduleOfInvestments} and \texttt{FinancialHighlights}, or
\texttt{CoverPage}) records which financial statement a fact belongs to. It is filled from the
\emph{authoritative} source --- the filing's own presentation linkbase (\texttt{*\_pre.xml}),
which groups concepts into statement sections --- so it is not a heuristic guess. Pass
\texttt{classify=true} to \texttt{facts(::Filing)}, \texttt{archive\_filings},
\texttt{to\_duckdb(f::Filing, db)}, or the interactive \texttt{select\_section} /
\texttt{select\_sections} to fill it (one extra fetch of the linkbase); concepts that
appear only in notes are left blank. The map is also available directly:

\sig{statement\_map(f::Filing) -> Dict\{String, String\}}

\paragraph{Multi-homed concepts.} A concept frequently belongs to \emph{more than one} statement:
\texttt{StockholdersEquity} appears in both the balance sheet and the statement of equity;
\texttt{NetIncomeLoss} appears in the income statement, the cash-flow reconciliation, the statement
of comprehensive income \emph{and} the equity roll-forward. The single \texttt{statement} column
records only the \emph{primary} (highest-priority) section, so every fact also carries a
\texttt{statements} column --- the JSON array of \emph{all} sections it belongs to, primary first (so
\texttt{statement} equals \texttt{statements[1]}). Without it, a literal \texttt{statement ==
"Equity"} filter would miss the \texttt{StockholdersEquity} totals (which are primarily the balance
sheet's). Accordingly the DuckDB \texttt{statement\_view} is \emph{membership-aware}: querying
\texttt{statement = "Equity"} returns every fact whose \texttt{statements} set contains \texttt{Equity}
--- the equity movements \emph{and} the multi-homed totals --- reconstructing the full statement.

\noindent Some filers (typically inline-only filings) ship \emph{no} loose presentation
linkbase. For those, classification falls back automatically to the SEC-generated
\texttt{FilingSummary.xml} and the rendered statement R-files --- present for \emph{every}
XBRL filing --- so \texttt{classify=true} works across filers rather than silently returning
nothing. Only the face statements are read: the classifier scores each report's role name, its
concepts and --- for the FilingSummary path --- its \texttt{LongName} category
(\texttt{Statement} vs \texttt{Disclosure}), with notes/details scoring as disqualifying. So a
generically named detail R-file (e.g. ``Comprehensive Income Statements (Detail)'', whose role still
embeds the word \emph{income statement}) is not mistaken for the statement itself. The map is empty
(and a warning is emitted) only if \emph{both} sources are missing.

\paragraph{Combined statements and the query-time resolver.} Classification reflects the filing's
\emph{actual} structure, not an idealised one --- and that occasionally bites. Some filers (many
foreign IFRS issuers, and a few domestic ones) present a single \emph{combined} ``Statement of Profit
or Loss and Other Comprehensive Income'' rather than a standalone income statement. For those, the
net-income line is correctly tagged \texttt{ComprehensiveIncome} --- that is the section the filing
puts it in --- so a literal filter for \texttt{statement == "IncomeStatement"} returns \emph{nothing}.
This is faithful, not a bug: silently re-tagging the fact would throw away the true fact that it lives
in the combined statement. To ask for the income statement \emph{regardless} of how a filer arranged
it, use the query-time resolver:

\sig{select\_statement(rows, statement) -> rows}

\noindent It returns the rows for \texttt{statement} directly when that section exists, and otherwise
falls back to the section that \emph{subsumes} it --- income $\to$ comprehensive income $\to$ equity ---
but only when that section actually carries the requested type's anchor concepts, so a pure
other-comprehensive-income section is never mistaken for the income statement. The same caveat applies
to the DuckDB \texttt{statement\_view}: querying \texttt{statement = "IncomeStatement"} for a
combined-statement filer returns no rows --- query \texttt{"ComprehensiveIncome"} instead, or run the
facts through \texttt{select\_statement} before loading them. The rule of thumb: the \texttt{statement}
column tells you \emph{where the filer put a number}; \texttt{select\_statement} tells you \emph{which
section answers your question}.

\paragraph{Statements filed as a note.} A related case: some filers present a whole statement --- most
often the statement of changes in equity --- \emph{only} as a note/detail disclosure, never as an XBRL
face section. Classification then leaves it blank, faithfully: the note's ``\ldots Details'' or
text-block role is not a face statement. To recover it on demand:

\sig{reconstruct\_from\_notes(f::Filing, statement) -> Vector\{FactRow\}}

\noindent finds the note role whose \emph{intent} is \texttt{statement} (a relaxed classification that
ignores the note/detail markers) and returns its facts re-tagged to \texttt{statement}. The result is
explicitly \textbf{reconstructed, not classified}: every row is marked \texttt{source\_selector =
"reconstructed:<role>"} (directly extracted facts carry an empty \texttt{source\_selector}), so the two
are never confused. It is opt-in --- ordinary \texttt{facts(f; classify=true)} stays faithful and never
includes these. The logic is jurisdiction-agnostic (a two-argument
\texttt{reconstruct\_from\_notes(pre\_xml, rows, statement)} takes any presentation linkbase and facts
table); the \texttt{Filing} method fetches both for an SEC filing.

\paragraph{Native labels.} The picker reads a fact's human-readable \texttt{label} off the
rendered table row; the browser-free extractor has no rendered page, so it reads the filing's
\emph{label linkbase} (\texttt{*\_lab.xml}) instead --- the authoritative \texttt{concept => label}
map (preferring the standard label, then terse, then verbose). Pass \texttt{labels=true} to
\texttt{facts(::Filing)}, \texttt{archive\_filings} or \texttt{to\_duckdb(f::Filing, db)} to fill
the \texttt{label} column (one extra fetch); without it native facts have an empty label. The map
is also available directly:

\sig{label\_map(f::Filing) -> Dict\{String, String\}}

\paragraph{Calculation relationships.} A filing's \emph{calculation linkbase}
(\texttt{*\_cal.xml}) is the arithmetic of each statement: which concepts sum into which, with
a signed weight (\texttt{+1} added, \texttt{-1} subtracted). \texttt{calculations} returns it as
a Tables.jl row table of \texttt{(statement, parent, child, weight)}.

\sig{calculations(f::Filing) -> Vector\{NamedTuple\}}

\noindent This is the authoritative \emph{how-it-rolls-up} structure --- use it to validate that
children sum to their parent (a data-quality check) or to read a line's contribution sign. It
does \emph{not} rewrite the stored fact values: those are XBRL-canonical and validated against the
SEC \texttt{companyfacts} API, and the printed sign of a line depends on its parent, which is
statement-specific. For Walmart, \texttt{us-gaap:OperatingIncomeLoss} =
\texttt{Revenues}\,(+1)\,$-$\,\texttt{CostOfRevenue}\,(-1)\,$-$\,\texttt{SGA}\,(-1), and the
children indeed sum to the reported operating income.

\paragraph{Concept standardization (pluggable).} Each fact row carries a
\texttt{standard\_concept}: the issuer's \texttt{concept} mapped to a common name so facts can be
grouped and compared across companies (the original \texttt{concept} and \texttt{label} are
always preserved --- the dual-label approach). A standardization mapping is, however, an
\emph{opinionated} artifact --- two distinct lines (Net sales vs.\ Total revenues) must not
collapse to one --- so EDGAR.jl ships \textbf{no} default: until you select a mapping with
\texttt{set\_standardizer}, \texttt{standard\_concept} is \texttt{nothing}.

\sig{standardize(concept) -> Union\{String, Nothing\}}
\sig{set\_standardizer(mapping)}

\noindent \texttt{mapping} is a \texttt{Dict} of \texttt{concept => standard\_concept}, a
function \texttt{concept -> Union\{String, Nothing\}}, or \texttt{:none}. Supply your own, or
select one of the mappings EDGAR.jl plans to offer, which fall into two kinds:
\begin{itemize}
\item \textbf{Reference} --- the us-gaap taxonomy and the SEC Financial Statement Data Sets give
  \emph{canonical} concepts and labels, but do \emph{not} collapse synonyms (two tags for
  ``revenue'' stay distinct);
\item \textbf{Community} --- a curated mapping that collapses synonyms for cross-company
  comparison, at the cost of being opinionated. EDGAR.jl \emph{bundles} one:
  \texttt{set\_standardizer(:edgartools)} loads the edgartools mapping (vendored, MIT-licensed
  and attributed --- see \texttt{src/data/edgartools\_concept\_mappings.NOTICE.md}), which maps,
  for example, \texttt{us-gaap:Revenues} to \texttt{Revenue} but
  \texttt{us-gaap:RevenueFromContractWithCustomer\ldots} to the \emph{distinct}
  \texttt{Contract Revenue}, so Net sales and Total revenues do not collapse.
\end{itemize}

\noindent The bundled edgartools mapping is also available directly --- as a plain
\texttt{concept => standard\_concept} \texttt{Dict} --- for inspection or to use as a base you
extend before passing to \texttt{set\_standardizer}:

\sig{edgartools\_mapping() -> Dict\{String, String\}}

\begin{lstlisting}
set_standardizer(:edgartools)               # the bundled community mapping (MIT, attributed)
standardize("us-gaap:Revenues")             # "Revenue"
standardize("us-gaap:RevenueFromContractWithCustomerExcludingAssessedTax")  # "Contract Revenue"
set_standardizer(Dict("xx:Foo" => "Bar"))   # or supply your own
set_standardizer(:none)                     # the default: no standardization
\end{lstlisting}

\noindent The bundled edgartools mapping is a point-in-time snapshot. Maintainers can re-vendor it
from upstream with \texttt{julia --project scripts/refresh\_standardizer.jl} (optionally passing a
tag or commit); the script validates the download as JSON before it replaces the bundled copy. See
\texttt{src/data/edgartools\_concept\_mappings.NOTICE.md}.

\paragraph{No extra sinks needed.} Parquet, CSV and SQLite are deliberately \emph{not}
separate exports. A fact table is a Tables.jl source, so \texttt{CSV.write(path, facts(sel))},
\texttt{Arrow.write(\ldots)} and \texttt{SQLite.load!(\ldots)} already take it in one line;
and once the facts are in DuckDB, \texttt{COPY facts TO 'f.parquet' (FORMAT PARQUET)} or
\texttt{ATTACH 'f.sqlite' (TYPE SQLITE)} export them with a single SQL statement. Wrapping
these would only add dependencies for no gain.

\subsubsection{company\_concept}

\sig{company\_concept(cik, taxonomy, tag)}

\noindent One XBRL concept over time for a single filer, from

\subsubsection{xbrl\_frames}

\sig{xbrl\_frames(taxonomy, tag, unit, period)}

\noindent One XBRL concept for one period across \emph{every} filer that reported
it, from \texttt{/api/xbrl/frames/} --- a cross-section rather than a time series. A
trailing \texttt{I} on \texttt{period} denotes an instant (e.g.\ \texttt{"CY2022Q4I"});
drop it for a duration (e.g.\ \texttt{"CY2022"}). Rows are under \texttt{.data}.

\begin{lstlisting}
facts   = company_facts("320193")
concept = company_concept("320193", "us-gaap", "NetIncomeLoss")
frame   = xbrl_frames("us-gaap", "Assets", "USD", "CY2022Q4I")

# Each observation carries: start, end, val, fy/fp, form, filed, accn, frame?
for o in filter(o -> o.form == "10-K", concept.units.USD)
    println(o.fy, "  ", o.val)
end
\end{lstlisting}

\subsubsection{set\_user\_agent}

\sig{set\_user\_agent(user\_agent)}

\noindent Set the SEC \texttt{User-Agent} for the current process from a single
\texttt{"name email"} string. It is validated (non-empty, contains an email) and
returned. A User-Agent is required before any request; without one EDGAR.jl stops
with a clear error \emph{before} contacting the SEC.

\subsubsection{get\_user\_agent}

\sig{get\_user\_agent()}

\noindent Return the \texttt{User-Agent} that will be sent, following the resolution
order: an explicit value from \texttt{set\_user\_agent}/\texttt{set\_config}, else the
\texttt{SEC\_USER\_AGENT} environment variable, else an error.

\subsubsection{persist\_user\_agent}

\sig{persist\_user\_agent(user\_agent; depot = first(DEPOT\_PATH))}

\noindent Validate and set the User-Agent for this session, \emph{and} write
\texttt{SEC\_USER\_AGENT} into the depot's \texttt{startup.jl} so every future Julia
session has it. \texttt{depot} selects which Julia depot's startup file to write.

\subsubsection{unpersist\_user\_agent}

\sig{unpersist\_user\_agent(; depot = first(DEPOT\_PATH))}

\noindent Remove the \texttt{SEC\_USER\_AGENT} line that \texttt{persist\_user\_agent}
wrote from the depot's \texttt{startup.jl}. Returns \texttt{true} if a line was
removed, \texttt{false} if there was nothing to remove.

\paragraph{Caching.}
Responses are cached so repeated calls in a session do not re-download the same
data. The \texttt{cache} keyword of \texttt{set\_config} selects where the cache
lives and how long it survives:

\begin{itemize}[nosep]
  \item \texttt{:temporary} (default) --- an ephemeral per-process directory,
    deleted when Julia exits. Nothing persists across sessions.
  \item \texttt{:persistent} --- kept in \texttt{\textasciitilde/.cache/EDGAR.jl}
    (or \texttt{XDG\_CACHE\_HOME}) across sessions; files older than
    \texttt{cache\_max\_age} are pruned automatically.
  \item \texttt{:off} --- no caching; every call re-fetches.
\end{itemize}

\noindent
Two \textbf{independent} time limits apply: \texttt{cache\_ttl} (default 24\,h) is
\emph{freshness} --- how long a cached response is reused before being re-fetched
(it deletes nothing); \texttt{cache\_max\_age} (default 7\,days) is \emph{retention}
--- how long a file is kept on disk before being pruned. \texttt{cache\_max\_size}
(default 10\,MB) caps the size of any single cached response.

\toprule
\textbf{Function} & \textbf{Purpose} \\
\midrule
\endhead
\texttt{set\_config(; cache, cache\_dir, cache\_ttl, cache\_max\_size,
  cache\_max\_age, host\_whitelist, allow\_file, user\_agent)} & Override the
  global runtime configuration (only the keywords you pass change). \\
\texttt{clean\_cache()} & Prune the cache on demand. \\

\subsubsection{set\_config}

\sig{set\_config(; cache, cache\_dir, cache\_ttl, cache\_max\_size, cache\_max\_age, host\_whitelist, allow\_file, user\_agent)}

\noindent Override the global runtime configuration, returning the updated
\texttt{EDGARConfig}. Only the keywords you pass are changed; the rest keep their
current values. \texttt{cache} selects the mode (\texttt{:temporary} / \texttt{:persistent}
/ \texttt{:off}); \texttt{cache\_ttl} sets freshness and \texttt{cache\_max\_age}
retention; \texttt{cache\_max\_size} caps a single response; \texttt{user\_agent} is an
alternative to \texttt{set\_user\_agent}.

\subsubsection{clean\_cache}

\sig{clean\_cache(max\_age\_seconds = CACHE\_TTL)}

\noindent Prune the on-disk cache on demand, deleting entries older than
\texttt{max\_age\_seconds} and returning the number removed.

\subsubsection{cache\_metrics}

\sig{cache\_metrics()}

\noindent Return a \texttt{Dict} of the session's cache counters --- hits, misses,
requests and bytes --- for inspecting cache effectiveness.

\subsubsection{cache\_path\_for}

\sig{cache\_path\_for(url)}

\noindent Return the on-disk cache path EDGAR.jl would use for \texttt{url}, whether
or not it is currently cached.

\begin{lstlisting}
set_config(cache = :persistent, cache_ttl = 3600)   # keep across sessions, 1h freshness

\subsection{Glossary}
\label{sec:glossary}

SEC filings and their structured data come with their own vocabulary.
The terms that matter here:

\begin{description}[leftmargin=1.4em,style=nextline,itemsep=2pt]
  \item[CIK] \emph{Central Index Key}, the SEC's unique ID for a filer,
    zero-padded to 10 digits (Apple $\to$ \texttt{0000320193}). Functions accept it
    as an integer or a string, with or without leading zeros (\texttt{320193}
    $\equiv$ \texttt{0000320193}). Look one up from a ticker or company name with
    \texttt{cik}.
  \item[Filing] a document submitted to the SEC, identified by an
    \textbf{accession number} and a \textbf{form type} (\texttt{10-K} annual
    report, \texttt{10-Q} quarterly, \texttt{8-K} current report, \dots). Get a filer's filings with \texttt{filings\_by\_cik}.
  \item[XBRL] \emph{eXtensible Business Reporting Language}, the machine-readable format companies tag their financial statements in; it is what makes the numbers queryable rather than locked inside a document. A pure XBRL filing is an XML document whose primary purpose is to represent structured facts,
  \begin{lstlisting}
<us-gaap:Assets contextRef="CurrentYear" unitRef="USD">
    1000000
</us-gaap:Assets>
\end{lstlisting}
A browser won't render this as a nice financial statement. It's intended for software.
\item[iXBRL] \emph{Inline-XBRL}: those same tags embedded directly inside the filing's human-readable HTML, so one document is both readable and
machine-parseable. The SEC has required it for financial statements since
around 2019. An iXBRL filing is an HTML document that looks like a normal report,
\begin{lstlisting}
<p>
Total assets were
<ix:nonFraction
    name="us-gaap:Assets"
    contextRef="CurrentYear"
    unitRef="USD">
    1000000
</ix:nonFraction>.
</p>
\end{lstlisting}
Historically, iXBRL was defined on top of XHTML (XML-compliant HTML), although modern tools often treat it as HTML with XML namespaces.
  \item[Fact] a single reported number: one \emph{concept}, for one
    \emph{period}, in one \emph{unit} (e.g.\ ``net income for FY2023 was
    96{,}995{,}000{,}000 USD''). \texttt{company\_facts} returns every fact a
    company has reported.
  \item[Concept] the \emph{name} of a reported line item, drawn from a taxonomy
    (e.g.\ \texttt{NetIncomeLoss}, \texttt{Assets}, \texttt{Revenues}) --- ``what is
    being measured''. \texttt{company\_concept} returns one concept over time.
  \item[Taxonomy] a dictionary that defines concepts. The big one is
    \textbf{\texttt{us-gaap}} (US accounting standards); \textbf{\texttt{dei}}
    (\emph{Document and Entity Information}) holds metadata about the filing
    itself; \textbf{\texttt{srt}} holds shared elements; \textbf{\texttt{ifrs-full}}
    is for filers reporting under IFRS; and each company has its own
    \textbf{extension} namespace (e.g.\ \texttt{aapl} for Apple-specific concepts).
  \item[Frame] a cross-section: one concept, one unit, one period, across
    \emph{every} filer that reported it. \texttt{xbrl\_frames} returns a frame.
  \item[Unit] the unit of measure of a fact: \texttt{USD}, \texttt{shares},
    \texttt{USD/shares} (per-share), \texttt{pure} (a ratio), \dots
  \item[Period] the time a fact covers, and how it is written. \textbf{Frames}
    use \emph{calendar} periods: \texttt{CY} + year, as in \texttt{CY2022} (a full
    calendar year), \texttt{CY2022Q4} (a quarter), or \texttt{CY2022Q4I} (a single
    instant --- note the trailing \texttt{I}). An \textbf{instant} is a point in
    time (balance-sheet items such as Assets, at quarter-end); a
    \textbf{duration} is a span (income-statement items such as Revenue). By
    contrast, \textbf{\texttt{FY}} in a company's own facts is its \emph{fiscal}
    year, which need not match the calendar (Apple's \texttt{FY2023} ended in
    September 2023).
\end{description}


\section{Architecture and Portability}
\label{sec:architecture}

EDGAR.jl is built around a deliberate split: a \textbf{jurisdiction-agnostic XBRL core} and a thin
\textbf{jurisdiction layer}. The core --- the regex-based extractor that resolves contexts, units
and facts from raw \texttt{<ix:nonFraction>} / \texttt{xbrli:context} / \texttt{xbrli:unit}, the
\texttt{Fact}/\texttt{Selection} model, the four export layers, the DuckDB warehouse, the browser
picker, and the linkbase \emph{parsers} --- depends on no SEC-specific assumption. It has been
validated unchanged against a non-SEC filing: the GLEIF 2024 annual report (an ESEF/IFRS report in
both inline \texttt{.xhtml} and classic \texttt{.xbrl} form) extracts identically from both
serializations, using the IFRS \texttt{ifrs-full} taxonomy, a \texttt{gleif} extension namespace,
and an LEI entity identifier --- none of which the core was written for.

What is genuinely SEC-specific is small: \emph{how} a filing is discovered and fetched, \emph{where}
its linkbases live, how its filer is \emph{identified}, the role-name \emph{vocabulary} used to
classify statements, and the SEC-only structured-data APIs. Table~\ref{tab:portability} shows where
the lines fall. SEC is the one jurisdiction implemented today; other jurisdictions (ESEF report
packages, Canada's SEDAR+) would supply only their slice of the jurisdiction column. Concretely, the
common parsing already lives in \texttt{extract\_xbrl.jl} and the SEC linkbase access in
\texttt{extract\_xbrl\_sec.jl}.

\centering

than a rewrite.

\subsection{Statement classification: two orthogonal axes}
\label{sec:classification-layers}

Deciding which financial statement a fact belongs to (\texttt{statement\_map}, \S\ref{sec:export-layers})
is the part most exposed to the messy diversity of real filings, so it is worth being explicit about
how that knowledge is layered. The robust approach --- a multi-signal scorer adapted from
\href{https://github.com/dgunning/edgartools}{edgartools} (MIT; see Acknowledgements) --- factors
along the \emph{taxonomy}, \emph{not} the jurisdiction. There are three layers
(Table~\ref{tab:classify-layers}).

\begin{table}[h]
\centering
\caption{The three layers of statement-classification knowledge.}
\label{tab:classify-layers}
\small
\begin{tabular}{@{}p{0.17\textwidth}p{0.60\textwidth}p{0.16\textwidth}@{}}
\toprule
\textbf{Layer} & \textbf{What it is} & \textbf{Shared by} \\
\midrule
\textbf{Engine logic} (common) & The cascade + scoring + validation \emph{meta-rules}: prefer a complete statement over a fragment, validate that essential concepts are present, reject the wrong type, penalize parentheticals, and combined-statement aliasing. Plus the structural semantics --- a balance sheet \emph{is} Assets + Liabilities + Equity; an income statement runs revenue $\to$ profit. & All taxonomies \& jurisdictions \\
\textbf{Taxonomy vocabulary} (per-taxonomy, often shared) & The concept anchors, concept patterns and role-name patterns. \texttt{ifrs-full} (profit-or-loss, financial position, changes in equity) is reused by ESEF, SEDAR+ (IFRS), Companies House (IFRS), DART (K-IFRS), MOPS (TW-IFRS) and EDINET (IFRS filers). \texttt{us-gaap} (tax-disclosure penalties, comprehensive-income variants, roll-forwards) is SEC. National GAAPs (JP \texttt{jpcrp}/\texttt{jppfs}, UK FRS~101/102, KR) are one regime each. & the taxonomy's users \\
\textbf{Jurisdiction adapter} (per-jurisdiction) & Discovery/fetch, linkbase location, identity (CIK / LEI / \ldots), the filing-package conventions (SEC's \texttt{FilingSummary.xml} + R-files; ESEF zip; EDINET/DART APIs), and which taxonomies the regime uses. & one jurisdiction each \\
\bottomrule
\end{tabular}
\end{table}

\noindent The edgartools scorer encodes roughly two dozen empirically-tuned, issue-driven rules
(each a real filing that broke a naive classifier, then a fix). Of those, about \emph{45\%} is common
engine logic, \emph{30\%} \texttt{us-gaap} data, and \emph{20\%} \texttt{ifrs-full} data --- and that
IFRS slice is reused across the jurisdictions above. Almost nothing in the engine is SEC-specific, so
adopting it front-loads both the engine \emph{and} the IFRS vocabulary: adding ESEF / Companies House
/ DART / MOPS / EDINET becomes mostly adapters plus reuse, with a new national-GAAP vocabulary only
where a regime actually uses one (chiefly Japan and the UK).

This refines the portability picture above into \textbf{two orthogonal axes}: a \emph{jurisdiction}
axis (\texttt{SEC <: Jurisdiction}, \ldots) handling fetch and identity, and a separate
\emph{taxonomy} axis (\texttt{us-gaap} / \texttt{ifrs-full} / \ldots\ vocabularies, auto-selected from
the concept prefixes a filing actually uses) handling classification. EDGAR.jl ships the
classification \emph{registry} today; the full per-request scoring cascade is a planned, test-anchored
port of the edgartools resolver. (This is a design direction and may change in detail.)

\section{Related Work and Comparison}
\label{sec:related-work}

EDGAR.jl enters a mature ecosystem of EDGAR/XBRL tooling, mostly in Python and R. It is
worth being explicit about where it follows the field, where it differs, and what it does
not (yet) do.

\paragraph{The landscape.} In Python, \href{https://github.com/dgunning/edgartools}{\texttt{edgartools}}
is the modern leader --- it turns filings into typed objects and is especially strong on XBRL
statements, with two ideas we admire: \emph{concept standardization} (mapping each issuer's tag
to a common concept for cross-company comparison) and \emph{statement stitching} (assembling a
multi-period statement across filings). It is, however, in-memory --- it exports to
\texttt{DataFrame}s rather than persisting a warehouse. \href{https://arelle.org/}{Arelle} is the
reference open-source XBRL engine --- the SEC's own EDGAR Renderer is built on it --- and its
\texttt{xbrlDB} plug-in persists facts using the canonical \emph{XBRL-US Public Database} schema.
Tools such as \texttt{sec-edgar-downloader} and \texttt{sec-downloader} fetch raw filings and
leave parsing to you. In R, \texttt{finstr} (on top of the \texttt{XBRL} package) organises
instances into income statements, balance sheets and cash-flow statements. Julia, by contrast,
has had no established EDGAR/XBRL library.

picker with provenance:} you select a region in the rendered filing, and every extracted fact
carries a \texttt{source\_selector} back to the exact DOM node --- a bridge between document
review, iXBRL semantics and relational analytics that the other tools, which extract purely
programmatically, do not offer. \emph{(2)~An embedded DuckDB warehouse:} zero-server and
single-file, where Arelle persists only to heavy databases and edgartools does not persist at
all --- and facts from any source share one table under a semantic key
(\S\ref{sec:export-layers}). \emph{(3)~A four-layer export} from a single selection --- lossless
iXBRL, Markdown, Facts JSON and DuckDB --- with provenance threaded throughout. And it is
Julia/Tables.jl-native, filling a gap in that ecosystem.

\paragraph{Authoritative enrichment.} A second distinction is \emph{where} EDGAR.jl's enrichment
comes from: a fact's statement is read from the filing's own presentation linkbase
(\texttt{statement\_map}, \S\ref{sec:export-layers}) and its roll-up arithmetic from the
calculation linkbase (\texttt{calculations}), rather than inferred --- and concept standardization
is pluggable (\texttt{set\_standardizer}), defaulting to \emph{none} and offering the bundled,
attributed edgartools mapping opt-in, instead of baking one opinionated mapping in.

\paragraph{What it does not (yet) do.} In fairness, the established tools still lead in breadth.
edgartools covers many more form types (Form 3/4/5, 13F, ADV, \dots) and now ships IFRS mappings,
while Arelle implements the full XBRL specification --- formula, dimensional validation, table
linkbases --- that EDGAR.jl does not. EDGAR.jl focuses on us-gaap financial-statement facts, and
its standardization is the bundled edgartools concept mapping rather than that project's newer,
data-driven one. The gaps an earlier draft named --- concept standardization, statement
classification, and the calculation linkbase --- are now closed
(\texttt{set\_standardizer(:edgartools)}, \texttt{statement\_map}, \texttt{calculations}).

\paragraph{Validation from the reference schema.} Reassuringly, the design we arrived at matches
the XBRL-US reference: it hashes each fact by its context so identical contexts collapse (our
semantic \texttt{fact\_key}), and assigns a fiscal year and period to every fact (our
\texttt{fy}/\texttt{fp}/\texttt{frame}). Where we diverge is deliberate --- the reference
normalises context, unit and dimension into separate tables for transactional Postgres, whereas
EDGAR.jl keeps a \emph{denormalised} wide facts table, which is what an analytical, columnar
store like DuckDB prefers.
