\begin{abstract}
\noindent
\textbf{EDGAR.jl} pulls company data from the U.S.\ Securities and Exchange
Commission's \href{https://www.sec.gov/search-filings/edgar-application-programming-interfaces}{EDGAR}
system into Julia: a filer's \emph{submission history}, the \emph{XBRL financial
facts} extracted from its filings, cross-filer \emph{frames}, and \emph{full-text
search} of filing contents. It talks to the public
\code{data.sec.gov} / \code{efts.sec.gov} / \code{sec.gov} endpoints with
\code{HTTP.jl} and \code{JSON3.jl}, parses HTML with \code{Gumbo.jl} and
\code{Cascadia.jl}, and caches responses so an interactive session does not
re-download the same data. No API key is required --- only a descriptive
\code{User-Agent} (see \S\ref{sec:ua}).
\end{abstract}

\section{Introduction}
\label{sec:intro}

EDGAR (\emph{Electronic Data Gathering, Analysis, and Retrieval}) is the SEC's
filing system. Every public company in the United States files its disclosures
there, and since 2009 the financial statements in those filings are tagged in
\textbf{XBRL}, which is what makes them machine-readable. The SEC publishes this
data through open JSON REST APIs that require no registration and no key.

EDGAR.jl wraps those endpoints behind a small, typed API and stops at plain Julia
values (\code{JSON3} objects and \code{NamedTuple}s), which you can pipe into
\code{Tables.jl}, \code{TimeSeries.jl}, or a database. It covers three kinds of
task:

\begin{itemize}[nosep]
  \item \textbf{Filings} --- list a filer's submissions, download a filing's
        documents, convert them to text, and extract sections such as
        \emph{Financial Statements} and \emph{Management's Discussion}.
  \item \textbf{XBRL financial data} --- a filer's full set of reported facts
        (\code{company\_facts}), one concept over time (\code{company\_concept}),
        or one concept across every filer for a period (\code{xbrl\_frames}).
  \item \textbf{Full-text search} --- search the contents of filings since 2001
        (\code{full\_text\_search}).
\end{itemize}

\noindent
The SEC asks for \textbf{fair use}: at most ten requests per second. EDGAR.jl's
cache (\S\ref{sec:caching}) helps you stay well under that.

\section{Installation}
\label{sec:install}

EDGAR.jl requires \textbf{Julia 1.12 or later}. It is not yet registered, so
install it straight from GitHub:

\begin{lstlisting}
using Pkg
Pkg.add(url = "https://github.com/Trumpingtons/EDGAR.jl")
\end{lstlisting}

\subsection{Setting a User-Agent}
\label{sec:ua}

The SEC's fair-access policy requires every request to identify you with a
\code{User-Agent} header containing your \textbf{name and a contact email}.
\textbf{Without it the SEC returns HTTP 403.} There is no key and no
registration --- just be honest about who is calling. Set yours once with a
single \code{"name email"} string:

\begin{lstlisting}
using EDGAR
set_user_agent("Jane Doe jane@example.com")
\end{lstlisting}

\noindent
\code{set\_user\_agent} validates the string (non-empty, contains an email) and
returns it. If you forget, EDGAR.jl stops with a clear error \emph{before} ever
contacting the SEC.

\subsection{Per Session User-Agent vs. Persistent}
\code{set\_user\_agent} (and \code{set\_config}) set the User-Agent only for the
\emph{current} Julia process --- a new REPL or notebook kernel starts without
one. The resolution order EDGAR.jl uses is:

\begin{enumerate}[nosep]
  \item an explicit value from \code{set\_user\_agent} / \code{set\_config}; else
  \item the \code{SEC\_USER\_AGENT} environment variable; else
  \item an error.
\end{enumerate}

\noindent
So to avoid setting it every session, define the environment variable once.
\code{persist\_user\_agent} writes it into your Julia startup file for you, which
every future Julia process (REPL, script, \emph{and} notebook kernel) reads:

\begin{lstlisting}
persist_user_agent("Jane Doe jane@example.com")  # writes ~/.julia/config/startup.jl
# unpersist_user_agent()                          # removes it again
\end{lstlisting}

\noindent
Equivalently, set \code{ENV["SEC\_USER\_AGENT"] = "Jane Doe jane@example.com"} in
\code{\textasciitilde/.julia/config/startup.jl} (or export it from your shell)
yourself. The User-Agent is \emph{not} a secret --- it is exactly the information
you are transmitting to the SEC --- so storing it in a startup file is fine.

\section{Quick Start}
\label{sec:quickstart}

\subsection{Searching}
\label{sec:qs-search}

Resolve a ticker to a CIK, then search the text of filings:

\begin{lstlisting}
using EDGAR
set_user_agent("Jane Doe jane@example.com")

# Ticker -> 10-digit, zero-padded CIK (exact match -> 0 or 1 row)
id = only(cik("AAPL"; by = :ticker)).cik   # "0000320193"

# Full-text search across filing contents (2001 onward), restricted to 10-K forms
res = filings_by_text("climate risk"; forms = "10-K")
println(res.hits.total.value, " matching filings")
for h in first(res.hits.hits, 5)
    s = h._source
    println(s.file_date, "  ", s.form, "  ", first(s.display_names))
end
\end{lstlisting}

\subsection{Financial Data}
\label{sec:qs-findata}

The three XBRL endpoints share the \code{us-gaap} vocabulary and return JSON:

\begin{lstlisting}
# Every XBRL fact a company has reported, in one document
facts = company_facts("320193")
gaap  = facts.facts[Symbol("us-gaap")]      # us-gaap has a hyphen: index by Symbol

# One concept over time (net income, grouped by unit)
ni = company_concept("320193", "us-gaap", "NetIncomeLoss")
println(ni.units.USD[end].val)

# The same concept across every filer for one period (a cross-section)
assets = xbrl_frames("us-gaap", "Assets", "USD", "CY2022Q4I")
println(length(assets.data), " filers reported Assets")
\end{lstlisting}

\noindent
A trailing \code{I} on the frame period (\code{"CY2022Q4I"}) denotes an
\emph{instant} (a balance-sheet point in time); drop it for a \emph{duration}
(a flow such as revenue). See the Glossary (\S\ref{sec:glossary}).

\subsection{Financial Reports}
\label{sec:qs-reports}

\emph{Structured extraction of financial statements from inline XBRL (iXBRL) is
planned and not yet available.} The SEC tags the financial statements directly
inside each filing's primary HTML document; a future release will parse those
\code{<ix:\dots>} tags into tidy tables.

In the meantime you can already work with the filing \emph{documents}: list a
filer's filings, download a filing, convert it to text, and pull named sections
(see \S\ref{sec:api-filings}). For the \emph{tagged numbers} that underlie the
statements today, use the XBRL data API above (\S\ref{sec:qs-findata}) --- the
SEC has already extracted those facts for you, so you rarely need to parse the
iXBRL by hand.

\section{Glossary}
\label{sec:glossary}

SEC filings and their structured (XBRL) data come with their own vocabulary.
The terms that matter here:

\begin{description}[leftmargin=1.4em,style=nextline,itemsep=2pt]
  \item[CIK] \emph{Central Index Key}, the SEC's unique ID for a filer,
    zero-padded to 10 digits (Apple $\to$ \code{0000320193}). Functions accept it
    as an integer or a string, with or without leading zeros (\code{320193}
    $\equiv$ \code{0000320193}). Look one up from a ticker or company name with
    \code{cik}.
  \item[Filing] a document submitted to the SEC, identified by an
    \textbf{accession number} and a \textbf{form type} (\code{10-K} annual
    report, \code{10-Q} quarterly, \code{8-K} current report, \dots). Get a
    filer's filings index with \code{fetch\_submissions}.
  \item[XBRL] \emph{eXtensible Business Reporting Language}, the machine-readable
    format companies tag their financial statements in; it is what makes the
    numbers queryable rather than locked inside a document.
  \item[iXBRL] \emph{Inline XBRL}: those same tags embedded directly inside the
    filing's human-readable HTML, so one document is both readable and
    machine-parseable. The SEC has required it for financial statements since
    around 2019.
  \item[Fact] a single reported number: one \emph{concept}, for one
    \emph{period}, in one \emph{unit} (e.g.\ ``net income for FY2023 was
    96{,}995{,}000{,}000 USD''). \code{company\_facts} returns every fact a
    company has reported.
  \item[Concept] the \emph{name} of a reported line item, drawn from a taxonomy
    (e.g.\ \code{NetIncomeLoss}, \code{Assets}, \code{Revenues}) --- ``what is
    being measured''. \code{company\_concept} returns one concept over time.
  \item[Taxonomy] a dictionary that defines concepts. The big one is
    \textbf{\code{us-gaap}} (US accounting standards); \textbf{\code{dei}}
    (\emph{Document and Entity Information}) holds metadata about the filing
    itself; \textbf{\code{srt}} holds shared elements; \textbf{\code{ifrs-full}}
    is for filers reporting under IFRS; and each company has its own
    \textbf{extension} namespace (e.g.\ \code{aapl} for Apple-specific concepts).
  \item[Frame] a cross-section: one concept, one unit, one period, across
    \emph{every} filer that reported it. \code{xbrl\_frames} returns a frame.
  \item[Unit] the unit of measure of a fact: \code{USD}, \code{shares},
    \code{USD/shares} (per-share), \code{pure} (a ratio), \dots
  \item[Period] the time a fact covers, and how it is written. \textbf{Frames}
    use \emph{calendar} periods: \code{CY} + year, as in \code{CY2022} (a full
    calendar year), \code{CY2022Q4} (a quarter), or \code{CY2022Q4I} (a single
    instant --- note the trailing \code{I}). An \textbf{instant} is a point in
    time (balance-sheet items such as Assets, at quarter-end); a
    \textbf{duration} is a span (income-statement items such as Revenue). By
    contrast, \textbf{\code{FY}} in a company's own facts is its \emph{fiscal}
    year, which need not match the calendar (Apple's \code{FY2023} ended in
    September 2023).
\end{description}

\section{API}
\label{sec:api}

CIKs are the SEC Central Index Key, zero-padded to 10 digits (Apple $\to$
\code{0000320193}). Any \code{cik} argument accepts an integer or a string, with
or without leading zeros (\code{320193}, \code{"320193"} and \code{"0000320193"}
are equivalent); functions normalize to the 10-digit form. Every data call goes
through the cached, User-Agent-aware request layer (\S\ref{sec:caching}).

\subsection{Searching and Company Lookup}
\label{sec:api-search}

\begin{longtable}{@{}p{0.46\textwidth}p{0.50\textwidth}@{}}

\midrule
\endhead

  \code{query} by company name (case-insensitive substring, the default), exact

\subsubsection{cik}

\medskip
\noindent\code{cik()}\\
\code{cik(query; by = :company | :ticker | :any)}
\par\medskip

\noindent A single lookup function that always returns a Tables.jl \emph{row
table}, which is a \texttt{Vector} of \texttt{NamedTuple}s with fields \texttt{company},
\texttt{ticker} and \texttt{cik}, with one record per ticker. \code{cik()} returns
every company; with a \code{query}, \code{by} selects which column to match:

\begin{itemize}[nosep]
  \item \code{:company} (the default) --- rows whose company name contains
    \code{query} (case-insensitive substring).
  \item \code{:ticker} --- \code{0} or \code{1} row matching the exact (case-insensitive) ticker.
  \item \code{:any} --- rows matching by company name \emph{or} ticker; handy when
    a short string like \code{"IBM"} could be either. Each row is tested once, so a
    row matching both columns appears only once.
\end{itemize}

\noindent The result type is the same for every form, so it stays type-stable, and
as a plain vector it indexes, filters and feeds straight into any tool that reads
tables, including \texttt{DataFrames}, \texttt{CSV} and \texttt{Parquet} files, or a \texttt{SQLite} database. A
\texttt{:company} query may yield
several rows, because (a) one company can have multiple tickers (e.g. share classes
\code{GOOGL}/\code{GOOG}), and (b) several companies may contain the query text in
their name.
\begin{lstlisting}

\subsubsection{filings\_by\_text}

\sig{filings\_by\_text(phrase; exact=true, forms=nothing, startdate=nothing, enddate=nothing, from=0)}

\noindent Also exported as \code{full\_text\_search} --- the SEC's own name for this
endpoint (EFTS) --- and pairs with \code{filings\_by\_cik}. The function
\texttt{filings\_by\_text} discovers \textit{filings} by their content (e.g. "which filings mention 'supply chain disruption'?"). Search the full text of SEC filings, everything filed from 2001 onward, for some \texttt{phrase} through the EDGAR
full-text search (EFTS) API, returning parsed JSON. \code{forms} may be a single string (\code{"10-K"})\footnote{Form 10-K is a detailed \textbf{annual} filing submitted by public companies to the SEC. It provides a comprehensive overview of the company's business activities, financial health, and key risks. The filing includes audited financial statements, management's discussion of financial results, information about the company's operations and strategy, disclosures on legal and regulatory issues, and an assessment of risk factors that could affect future performance.} or a collection; \code{startdate} and \code{enddate}
are \code{"YYYY-MM-DD"} strings; and \code{from} defines the offset used by EFTS to return a fixed page of 100 results per
request. By default the \texttt{phrase} is matched \textbf{exactly} (it is quoted
for you), so \code{"supply chain disruption"} finds only filings with those three
words adjacent; pass \code{exact = false} to match the words loosely instead.
Results come back ranked by relevance, not by date. To search by filer instead of
text, use \code{filings\_by\_cik} (below).

\begin{lstlisting}
julia> res = filings_by_text("supply chain disruption"; forms = "10-K", startdate = "2026-01-01")
JSON3.Object{Vector{UInt8}, Vector{UInt64}} with 6 entries:
  :took         => 2600
  :timed_out    => false
  :_shards      => {...
  :hits         => {...
  :aggregations => {...
  :query        => {...
\end{lstlisting}

The output looks cryptic because the SEC runs this on \texttt{Elasticsearch}, and the function returns its raw response envelope. 

\medskip
\noindent
\begin{tabular}{@{}p{0.26\textwidth}p{0.66\textwidth}@{}}
\toprule
\textbf{Key} & \textbf{Meaning} \\
\midrule
\code{:took}         & Milliseconds the search engine spent. \\
\code{:timed\_out}   & Whether the search timed out. \\
\code{:\_shards}     & Elasticsearch internal shard stats --- ignore. \\
\code{:hits}         & \textbf{Results}, the only key we usually need. \\

\subsubsection{filings\_by\_cik}

\sig{filings\_by\_cik(cik; forms=nothing, startdate=nothing, enddate=nothing, from=0)}

\noindent List a single filer's filings (2001 onward) through the same EFTS
endpoint, but keyed by its \texttt{cik} (an integer or string, padded or not) using
EDGAR's \emph{entity} filter --- the web form's company/CIK field. This is a true
``filed by this filer'' query, unlike full-text searching for the CIK as document
text, which also matches filings that merely \emph{mention} it (e.g.\ a fund listing
the company among its holdings). \code{forms}, the dates, \code{from} and the
\code{.hits} result layout are exactly as for \code{full\_text\_search}. 

\begin{lstlisting}
julia> res = filings_by_cik(320193; forms = "8-K", startdate = "2026-01-01")
JSON3.Object{Vector{UInt8}, Vector{UInt64}} with 6 entries:
  :took         => 3648
  :timed_out    => false
  :_shards      => {...
  :hits         => {...
  :aggregations => {...
  :query        => {...
\end{lstlisting}

Because no
text is matched, results come back \textbf{newest-first} and carry no relevance
score --- \code{res.hits.max\_score} and each \code{h.\_score} are \code{nothing}.

julia> res.hits.total.value
5
\end{lstlisting}

\subsection{Filings: List, Download, Extract}
\label{sec:api-filings}

\begin{longtable}{@{}p{0.46\textwidth}p{0.50\textwidth}@{}}
\toprule
\textbf{Function} & \textbf{Purpose} \\
\midrule
\endhead
\code{fetch\_submissions(cik)} & Full submissions JSON for a filer (company
  metadata + recent filings index under \code{.filings.recent}). \\
\code{download\_filing(cik, accession; destdir)} & Download a filing's primary
  document into \code{destdir}; tries common URL patterns, saving the first that
  returns HTTP 200. \\
\code{parse\_filing(path)} & Convert a filing's HTML to text (\code{Gumbo} +
  \code{Cascadia}). \\
\code{extract\_section(html, names; max\_chars)} & Pull named sections (e.g.\
  \code{"Item 7"}, \code{"Management's Discussion"}) from filing HTML/text. \\
\code{save\_filing(text, metadata; outdir)} & Write extracted text + metadata to
  disk. \\
\bottomrule
\end{longtable}

\subsubsection{fetch\_submissions}

\sig{fetch\_submissions(cik)}

\noindent Fetch a filer's complete submissions document from
\code{data.sec.gov/submissions/} --- company metadata plus its recent-filings
index --- as parsed JSON. \code{cik} may be an integer or a string, with or
without leading zeros. The recent filings sit under \code{.filings.recent} as
index-aligned column arrays (\code{accessionNumber}, \code{form},
\code{filingDate}, \code{primaryDocument}, \code{isXBRL}, \dots); e.g.\ the most
recent filing's accession is \code{fetch\_submissions(cik).filings.recent.accessionNumber[1]}.

\subsubsection{download\_filing}

\sig{download\_filing(cik, accession; destdir = ".")}

\noindent Download a filing's document from the EDGAR \code{Archives} into
\code{destdir} (created if needed) and return the local file path. The accession
number may be given with or without dashes. Several common document/index URL
patterns are tried in turn; the first that responds with HTTP 200 is saved, and
the function throws if none succeed.

\subsubsection{parse\_filing}

\sig{parse\_filing(path)}

\noindent Convert the filing HTML at \code{path} to plain text, using \code{Gumbo}
and \code{Cascadia} to strip markup. The extraction helpers operate on this text.

\subsubsection{extract\_section}

\sig{extract\_section(html, names; max\_chars = 200\_000, base\_path = nothing)}

\noindent Pull named sections from filing HTML/text by heuristic,
case-insensitive matching --- e.g.\ \code{"Item 7"} or
\code{"Management's Discussion"}. Returns a \code{Dict} mapping each requested name
to its extracted text; \code{max\_chars} caps the length kept per section.

\subsubsection{save\_filing}

\sig{save\_filing(text, metadata; outdir = "out")}

\noindent Write extracted \code{text} together with its \code{metadata} to disk
under \code{outdir}, returning the path written.

\begin{lstlisting}
# List a company's recent filings, then download the most recent one
r = fetch_submissions("0000320193").filings.recent
for i in 1:5
    println(r.filingDate[i], "  ", r.form[i], "  ", r.accessionNumber[i])
end

path = download_filing("0000320193", r.accessionNumber[1]; destdir = "filings")

# Convert to text and extract named sections
html     = parse_filing(path)
sections = extract_section(html, ["Item 7", "Management's Discussion"])
println(get(sections, "Item 7", "(not found)"))
\end{lstlisting}

\subsection{Financial Data (XBRL)}
\label{sec:api-xbrl}

\begin{longtable}{@{}p{0.46\textwidth}p{0.50\textwidth}@{}}
\toprule
\textbf{Function} & \textbf{Purpose} \\
\midrule
\endhead
\code{company\_facts(cik)} & Every XBRL fact a filer has reported, in one
  (large) document. \\
\code{company\_concept(cik, taxonomy, tag)} & One concept over time for a single
  filer (lighter than \code{company\_facts}). \\

\subsubsection{company\_facts}

\sig{company\_facts(cik)}

\noindent Every XBRL fact a company has ever reported, in a single (often several
megabyte) document from the \code{/api/xbrl/companyfacts/} endpoint, as parsed
JSON. \code{cik} may be an integer or a string, with or without leading zeros.
Facts are grouped by taxonomy under \code{.facts} (e.g.\ \code{:dei},
\code{Symbol("us-gaap")}).

\subsubsection{company\_concept}

\sig{company\_concept(cik, taxonomy, tag)}

\noindent One XBRL concept over time for a single filer, from
\code{/api/xbrl/companyconcept/} --- lighter than \code{company\_facts} when you
want a single line item. Observations live under \code{.units} keyed by unit (e.g.
\code{.units.USD}), each carrying \code{start}/\code{end}, \code{val}, \code{fy},
\code{fp}, \code{form} and \code{filed}.

\subsubsection{xbrl\_frames}

\sig{xbrl\_frames(taxonomy, tag, unit, period)}

\noindent One XBRL concept for one period across \emph{every} filer that reported
it, from \code{/api/xbrl/frames/} --- a cross-section rather than a time series. A
trailing \code{I} on \code{period} denotes an instant (e.g.\ \code{"CY2022Q4I"});
drop it for a duration (e.g.\ \code{"CY2022"}). Rows are under \code{.data}.

\begin{lstlisting}
facts   = company_facts("320193")
concept = company_concept("320193", "us-gaap", "NetIncomeLoss")
frame   = xbrl_frames("us-gaap", "Assets", "USD", "CY2022Q4I")

# Each observation carries: start, end, val, fy/fp, form, filed, accn, frame?
for o in filter(o -> o.form == "10-K", concept.units.USD)
    println(o.fy, "  ", o.val)
end
\end{lstlisting}

\subsection{Credentials and Caching}
\label{sec:caching}

\paragraph{Credentials (the User-Agent).}
There is no API key; the only credential is the SEC \code{User-Agent}
(\S\ref{sec:ua}).

\begin{longtable}{@{}p{0.46\textwidth}p{0.50\textwidth}@{}}
\toprule
\textbf{Function} & \textbf{Purpose} \\
\midrule
\endhead
\code{set\_user\_agent(user\_agent)} & Set the User-Agent from a single
  \code{"name email"} string (validated). \\
\code{get\_user\_agent()} & Return the User-Agent that will be sent (explicit
  value $\to$ \code{SEC\_USER\_AGENT} env var $\to$ error). \\
\code{persist\_user\_agent(user\_agent; depot)} & Validate, set for this session,
  and write \code{SEC\_USER\_AGENT} into the depot's \code{startup.jl} so every
  future session has it. \\
\code{unpersist\_user\_agent(; depot)} & Remove the line that
  \code{persist\_user\_agent} wrote. \\
\bottomrule
\end{longtable}

\subsubsection{set\_user\_agent}

\sig{set\_user\_agent(user\_agent)}

\noindent Set the SEC \code{User-Agent} for the current process from a single
\code{"name email"} string. It is validated (non-empty, contains an email) and
returned. A User-Agent is required before any request; without one EDGAR.jl stops
with a clear error \emph{before} contacting the SEC.

\subsubsection{get\_user\_agent}

\sig{get\_user\_agent()}

\noindent Return the \code{User-Agent} that will be sent, following the resolution
order: an explicit value from \code{set\_user\_agent}/\code{set\_config}, else the
\code{SEC\_USER\_AGENT} environment variable, else an error.

\subsubsection{persist\_user\_agent}

\sig{persist\_user\_agent(user\_agent; depot = first(DEPOT\_PATH))}

\noindent Validate and set the User-Agent for this session, \emph{and} write
\code{SEC\_USER\_AGENT} into the depot's \code{startup.jl} so every future Julia
session has it. \code{depot} selects which Julia depot's startup file to write.

\subsubsection{unpersist\_user\_agent}

\sig{unpersist\_user\_agent(; depot = first(DEPOT\_PATH))}

\noindent Remove the \code{SEC\_USER\_AGENT} line that \code{persist\_user\_agent}
wrote from the depot's \code{startup.jl}. Returns \code{true} if a line was
removed, \code{false} if there was nothing to remove.

\paragraph{Caching.}
Responses are cached so repeated calls in a session do not re-download the same
data. The \code{cache} keyword of \code{set\_config} selects where the cache
lives and how long it survives:

\begin{itemize}[nosep]
  \item \code{:temporary} (default) --- an ephemeral per-process directory,
    deleted when Julia exits. Nothing persists across sessions.
  \item \code{:persistent} --- kept in \code{\textasciitilde/.cache/EDGAR.jl}
    (or \code{XDG\_CACHE\_HOME}) across sessions; files older than
    \code{cache\_max\_age} are pruned automatically.
  \item \code{:off} --- no caching; every call re-fetches.
\end{itemize}

\noindent
Two \textbf{independent} time limits apply: \code{cache\_ttl} (default 24\,h) is
\emph{freshness} --- how long a cached response is reused before being re-fetched
(it deletes nothing); \code{cache\_max\_age} (default 7\,days) is \emph{retention}
--- how long a file is kept on disk before being pruned. \code{cache\_max\_size}
(default 10\,MB) caps the size of any single cached response.

\begin{longtable}{@{}p{0.46\textwidth}p{0.50\textwidth}@{}}
\toprule
\textbf{Function} & \textbf{Purpose} \\
\midrule
\endhead
\code{set\_config(; cache, cache\_dir, cache\_ttl, cache\_max\_size,
  cache\_max\_age, host\_whitelist, allow\_file, user\_agent)} & Override the
  global runtime configuration (only the keywords you pass change). \\
\code{clean\_cache()} & Prune the cache on demand. \\
\code{cache\_metrics()} & Inspect hit/miss/request/byte counters. \\
\code{cache\_path\_for(url)} & The on-disk cache path for a given URL. \\
\bottomrule
\end{longtable}

\subsubsection{set\_config}

\sig{set\_config(; cache, cache\_dir, cache\_ttl, cache\_max\_size, cache\_max\_age, host\_whitelist, allow\_file, user\_agent)}

\noindent Override the global runtime configuration, returning the updated
\code{EDGARConfig}. Only the keywords you pass are changed; the rest keep their
current values. \code{cache} selects the mode (\code{:temporary} / \code{:persistent}
/ \code{:off}); \code{cache\_ttl} sets freshness and \code{cache\_max\_age}
retention; \code{cache\_max\_size} caps a single response; \code{user\_agent} is an
alternative to \code{set\_user\_agent}.

\subsubsection{clean\_cache}

\sig{clean\_cache(max\_age\_seconds = CACHE\_TTL)}

\noindent Prune the on-disk cache on demand, deleting entries older than
\code{max\_age\_seconds} and returning the number removed.

\subsubsection{cache\_metrics}

\sig{cache\_metrics()}

\noindent Return a \code{Dict} of the session's cache counters --- hits, misses,
requests and bytes --- for inspecting cache effectiveness.

\subsubsection{cache\_path\_for}

\sig{cache\_path\_for(url)}

\noindent Return the on-disk cache path EDGAR.jl would use for \code{url}, whether
or not it is currently cached.

\begin{lstlisting}
set_config(cache = :persistent, cache_ttl = 3600)   # keep across sessions, 1h freshness
m = cache_metrics()                                  # (; hits, misses, requests, ...)
clean_cache()
\end{lstlisting}

\subsection{Low-Level Requests}
\label{sec:api-lowlevel}

\subsubsection{fetch\_url}

\sig{fetch\_url(url; use\_cache = true, timeout = 15, allow\_file = false)}

\noindent The primitive every function above is built on: it returns the raw
response body as bytes (or \code{nothing} on failure), with the configured
\code{User-Agent} and the cache applied. Use it as an escape hatch to reach SEC
endpoints EDGAR.jl does not yet wrap, then parse the bytes yourself (e.g.\ with
\code{JSON3.read}).

\begin{lstlisting}
body = fetch_url("https://data.sec.gov/submissions/CIK0000320193.json")
data = JSON3.read(body)
\end{lstlisting}
